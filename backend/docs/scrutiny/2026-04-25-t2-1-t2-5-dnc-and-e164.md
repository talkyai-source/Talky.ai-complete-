## T2.1 — DNC list service + admin CRUD

### What was broken

`CallGuard._check_dnc` at `backend/app/domain/services/call_guard.py:717`
queries `dnc_entries` on every origination. The table exists (day7 +
20260402 migrations) but there was **no management surface** —
tenants couldn't add numbers, couldn't bulk-import, couldn't record
in-call opt-outs, couldn't even inspect their list. The check
silently passed everything because the table was always empty.

### What shipped

**`backend/app/domain/services/dnc_service.py`** — thin service
wrapping the existing table.

- `add(tenant_id, e164, source, reason=None, expires_at=None)` —
  idempotent insert. Normalises the input via libphonenumber
  before write.
- `add_caller_opt_out(tenant_id, e164, call_id)` — dedicated
  shortcut for the in-call opt-out path. Records
  `source='caller_opt_out'` with the `call_id` in the reason for
  audit.
- `bulk_import(tenant_id, numbers, source)` — per-row
  accepted/skipped/invalid counts.
- `is_on_dnc(tenant_id, e164)` — pre-flight check callers (UI,
  campaign loader) can run BEFORE the call hits CallGuard.
- `list_for_tenant(tenant_id, include_global=False)`,
  `remove(tenant_id, entry_id)` — standard CRUD.

**Source taxonomy** as module constants so call sites stay
consistent:
`caller_opt_out`, `manual_admin`, `ftc_national`,
`regulator_complaint`, `bulk_import`. Free-text in the DB so
future sources don't require a migration.

**`backend/app/api/v1/endpoints/dnc.py`** — six endpoints:

| Method | Path | Purpose |
|--------|------|---------|
| GET    | `/api/v1/dnc/` | List current tenant's DNC entries (with `?include_global=true` to also surface FTC-style cross-tenant entries) |
| POST   | `/api/v1/dnc/` | Add a single number (`manual_admin` default) |
| POST   | `/api/v1/dnc/bulk-import` | Upload many (up to 10 000); response splits accepted/skipped/invalid |
| POST   | `/api/v1/dnc/caller-opt-out` | Record an in-call opt-out (voice pipeline calls this) |
| GET    | `/api/v1/dnc/check?e164=…` | Pre-flight check (includes global entries) |
| DELETE | `/api/v1/dnc/{id}` | Remove — tenant-scoped only; global entries stay DBA-managed |

### Design choices

- **Normalise once on write.** Stored numbers are always E.164 so
  the hot-path CallGuard query can do a plain equality match on
  `normalized_number`.
- **Global entries with `tenant_id=NULL`** apply cross-tenant
  (FTC national DNC, for instance). The existing CallGuard query
  already matches both tenant-scoped and global rows; the new
  endpoints never let a tenant write with `tenant_id=NULL`.
- **Expiry supported** for time-bounded complaints. `NULL` means
  permanent.
- **No NLU in this sprint.** Actual "stop calling me" detection
  from caller audio is a voice-pipeline change. The
  `/caller-opt-out` endpoint is the integration point — when the
  pipeline detects the phrase (or a DTMF opt-out), it POSTs.

### What is NOT in this sprint

- **FTC national DNC list ingestion.** Operationally requires a
  paid subscription and agreement with the FTC. The service and
  source taxonomy are ready (`ftc_national`); wiring a periodic
  loader is a separate piece of work documented in the plan.
- **Voice-pipeline opt-out detection.** Needs NLU + DTMF handling
  in the live pipeline. Endpoint is ready to receive the call
  when that ships.

### Tests (15 new)

`backend/tests/unit/test_dnc_service.py`:
- E.164 normalisation across formats (parenthesised, dashed,
  dotted, spaced, already-E.164, UK international).
- `add()` idempotency — same number twice refreshes, doesn't
  duplicate.
- `add()` rejects empty numbers (explicit ValueError).
- `add_caller_opt_out()` sets `source=caller_opt_out` and embeds
  the `call_id` in the reason.
- `bulk_import()` splits accepted / skipped / invalid correctly.
- `is_on_dnc()` matches both tenant-scoped AND global entries;
  normalises the query side.
- `remove()` is tenant-scoped — can't delete another tenant's
  entry.
- `list_for_tenant()` excludes global by default, includes them
  when asked.

## T2.5 — Internationalised E.164 normalisation

### What was broken

`normalize_phone_number` in `backend/app/api/v1/endpoints/campaigns.py:454`
hard-defaulted any unprefixed number to `+1`. A UK number typed
`020 7946 0958` became `+1 020 7946 0958` — nonsense. Tenants
running non-US campaigns had to pre-format every contact.

### What shipped

Same function, now libphonenumber-aware:

```python
def normalize_phone_number(phone: str, default_country: str = "US") -> str:
    ...
```

- **libphonenumber first.** Parse with the supplied
  `default_country` (ISO-3166 alpha-2). If valid, format as E.164
  and return.
- **Legacy heuristic as fallback.** If libphonenumber rejects
  (invalid country, truncated number, lib unavailable), fall
  through to the old US-centric path. Preserves behaviour on
  machines that haven't pip-installed phonenumbers yet.
- **Per-campaign default country.** The contacts endpoint reads
  `campaign.script_config.default_country_code` (or the nested
  `campaign_slots.default_country_code`) so each campaign can
  declare its own default. Falls back to `US` when not configured
  — exact pre-T2.5 behaviour for existing campaigns.

### Tests (13 new)

`backend/tests/unit/test_phone_normalization_intl.py`:

US default (back-compat):
- 10-digit / formatted / pre-plus / 11-digit-with-1 all produce
  the same +1… output as before.

International — the T2.5 win:
- UK 020 7946 0958 + `default_country="GB"` → `+44…`.
- `+442079460958` + `default_country="US"` still → `+44…`
  (explicit plus overrides the default).
- German `030 2345 6789` + `default_country="DE"` → `+49…`.
- Australian `02 9374 4000` + `default_country="AU"` → `+61…`.

Edge cases:
- SIP extension (`1001`) passes through unchanged.
- Empty / too-short / too-long raise ValueError.
- Garbage country code (`"XX"`) falls back to legacy path instead
  of crashing.
