## DOC-03 · C++ voice gateway deep dive

**Maps to:** original task 3 · **Est:** 3h · **Status:** ⬜ · **Depends on:** TKT-001

### Goal — done means
`docs/v2/03-voice-gateway.md`: why this component exists in C++, what it does, how it connects to the
Python pipeline, its RTP timing logic, how to build it, how to restart it.

### ⚠️ Scope forks on TKT-001's answer
The original task assumes this is a running production component. That is not established. See
revision **R-4**:

- `services/voice-gateway-cpp/README.md` self-describes as *"Scaffold created for frozen Talk-Leee
  day plan execution"*.
- **No systemd unit for it exists in the repo** — the seven units present are api, dialer, voice,
  reminder, cleanup, healthwatch, trunk-status.
- Yet `deploy_to_server.sh` prints the status of a unit named `talky-voice-gateway`, and the server's
  working tree carries a modified compiled binary at `services/voice-gateway-cpp/build/voice_gateway`.

**Branch A — it IS running in production.** Write the full doc as specified, **and raise the
unversioned-unit-file issue as a production risk**: an unversioned systemd unit is one server rebuild
away from being permanently lost. Getting that file into `backend/systemd/` is a same-week action,
not a v2 idea.

**Branch B — it is NOT running.** DOC-03 becomes a shorter, honest doc: what it is, what it was built
for, why it is not deployed, what state it is in, and what adopting it would require. Roughly half the
budget; redirect the rest to DOC-05, which is the harder doc.

**Decide the branch from `00-ground-truth.md` before writing a word.**

### Plan (Branch A; trim for Branch B)
1. **Why C++.** The real reason — RTP is a 20ms hard-real-time loop and Python's GC pauses and GIL
   make jitter-free pacing unreliable at concurrency. Say it plainly; it is the first question anyone
   asks about a C++ component in a Python codebase.
2. **What it does.** From `README.md` and source: RTP send/receive with 20ms pacing, echo mode,
   `/health` + `/stats`, session start/stop/stats control API, no-RTP watchdog, jitter buffer. Codec
   locked to `pcmu`/20ms.
3. **Source map** — `src/session.cpp` (~1,753), `src/http_server.cpp` (~1,731),
   `src/session_registry.cpp` (~438), `src/rtp_packet.cpp` (~154), `src/main.cpp` (~135); headers
   under `include/voice_gateway/`. ~6,600 lines total including tests.
4. **Connection to Python** — the control API surface and where the Python side calls it.
5. **RTP timing** — the pacing loop, the jitter buffer, the no-RTP watchdog and what trips it.
6. **Build** — CMake ≥3.16, C++20. Document the ASan/UBSan and TSan variants, and the CTest targets
   (`voice_gateway_tests` 120s, `voice_gateway_concurrency_tests` 300s serial,
   `voice_gateway_fixes_tests` 180s serial) plus `tests/run_gate.sh`.
7. **Restart** — the exact command, and how to confirm it came back.
8. Note that `build/` and `build-asan/` artifacts are committed to the tree — a finding for DOC-09.

### Do
- Answer "why C++" first and properly. It is the question the task actually asks.
- Document the build on a clean checkout. A build procedure that only works on the machine that has
  built it before is not documented.

### Don't
- ❌ Do not write Branch A's doc without confirming from TKT-001 that it is running.
- ❌ Do not rebuild or restart the gateway on production to "check". Read-only.
- ❌ Do not omit the unversioned-unit-file risk if Branch A applies. It is the most important thing
  this ticket can surface.

## DOC-04 · Telephony stack — live vs scaffold

**Maps to:** original task 4 · **Est:** 4h · **Status:** ⬜ · **Depends on:** TKT-001

### Goal — done means
`docs/v2/04-telephony.md`: which telephony components are actually running, which exist only as
design work, why both exist, and how to read SIP logs.

### ⚠️ The original task's framing needs correcting
Task 4 asks about *"Asterisk, FreeSWITCH, OpenSIPS, Kamailio, RTPengine — which is active vs standby"*.
The evidence (revision **R-5**) suggests production runs **bare-metal Asterisk only**:

- `setup-asterisk.sh` provisions Asterisk via `apt install`, writing `/etc/asterisk/{http,ari,rtp,pjsip,extensions}.conf`
  with a direct PJSIP trunk to `sip3.blazedigitel.com`. It does **not** use `telephony/asterisk/Dockerfile`.
- `telephony/README.md` claims OpenSIPS is the active SIP edge — **this conflicts with the above** and
  must be resolved by TKT-001's process list, not by trusting either document.
- `telephony/freeswitch/conf/**` and `telephony/modules/*` are almost entirely `.gitkeep` placeholders.
- The `telephony/` tree is a genuine three-phase design and validation exercise with a signoff doc and
  hundreds of evidence artifacts. That is valuable work. It is not the same as running infrastructure.

**The doc's job is to state which is which, unambiguously, so nobody debugs a live incident against a
component that is not running.** That single sentence is worth more than the rest of the document.

### Plan
1. Open with a **process-level inventory from TKT-001**: what is running, right now, on the box.
2. For each of the five components: is it live, standby, or design-only? What does it handle when
   live? Where is its config — and note where two configs exist for the same component
   (`setup-asterisk.sh` writes `/etc/asterisk/*` while `telephony/asterisk/conf/*` exists in-repo;
   which one is authoritative?).
3. Document the **live** signal path end to end: carrier → PJSIP trunk → Asterisk → ARI → backend.
4. Trunks and caller-IDs: how a campaign maps to a trunk, where per-campaign trunk snapshots live, and
   the caller-ID verification requirement. Note the known trap: `+442046132301` is the real caller-ID
   for `blaze-pool-150004` — a one-digit-short value has already caused a live failure.
5. **How to read SIP logs** — `pjsip show registrations`, `pjsip set logger on`, journald for the
   ARI-side view. Include a worked example of a real failed registration.
6. Explain honestly why the scaffold exists: it is a planned migration to a proper SBC-fronted
   architecture, executed and validated, awaiting a deployment decision.
7. Cross-reference `telephony/README.md` and mark its "active runtime" claim corrected or confirmed.

### Do
- Lead with what is running. Everything else is context.
- Give a real worked log example. "How to read SIP logs" without a sample is not a how-to.
- Be respectful about the scaffold. It is thorough work; it is just not deployed.

### Don't
- ❌ Do not describe OpenSIPS/Kamailio/RTPengine/FreeSWITCH as active without process evidence.
- ❌ Do not enable `pjsip set logger on` on production and leave it on — it is verbose and will flood
  the journal. If you turn it on to capture an example, turn it off, and say so in the doc.
- ❌ Do not modify any Asterisk config.
- ❌ Do not print trunk credentials. `setup-asterisk.sh` contains hardcoded carrier credentials —
  **that is itself a finding for DOC-09 and a security item**, and the doc must reference the file
  without reproducing the secret.
