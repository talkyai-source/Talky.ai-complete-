## VID-14 · Live call trace

**Maps to:** original task 14 · **Est:** 4h · **Status:** ⬜ · **Script:** DOC-01

### Goal — done means
A recording that triggers a **real outbound call** and walks through the logs and the code in real
time, showing exactly what happens at each stage from the moment the dialer picks up the job.

### Why this is the most valuable recording in the set
Every other video explains structure. This one shows the system *working*, with real timing and real
log lines. It is what a new engineer will re-watch.

### Plan
1. **Prepare before recording.** Terminal panes laid out: dialer worker journal, voice worker journal,
   API journal, and an editor with DOC-01's flow diagram open. Trigger command ready.
2. Start recording. Walk the diagram for two minutes before dialling so the viewer knows what to watch for.
3. Trigger **one** call to the number confirmed with the user.
4. Narrate the stages live as the log lines appear:
   - dialer picks up the job → call-guard evaluation → origination request
   - `telephony_bridge.make_call` → guard checks → ARI originate
   - ringing → answer → session creation → pipeline start
   - the greeting — **point out `interrupted=False`**, and explain what it means and what it looked
     like when this was broken
   - the first caller turn: STT chunk → turn end → LLM → TTS → audio back
   - a deliberate barge-in — interrupt the agent mid-sentence and show it yield
   - hangup → call ended → transcript persisted → summary generated
5. After the call, replay the log window at a readable pace and map each line to the code, jumping to
   the module as you go.
6. Note explicitly that the summary comes from `call_summary/summarizer.py` — **and that
   `post_call_analyzer.py`, despite its name, is not in this path.** Someone will otherwise search for
   the analyzer and read the wrong file.

### Do
- Do the barge-in. It demonstrates two fixes at once — the greeting echo-immunity and that immunity
  not having become deafness.
- Let silences sit. Real latency is the point; editing it out teaches the wrong expectation.
- If something unexpected happens, narrate it and diagnose it live. That is more useful than a clean take.

### Don't
- ❌ Do not start a campaign. **One** call, to one confirmed number. The allstate list holds ~691 real
  leads.
- ❌ Do not dial a real customer for a demo. Use the number the user names.
- ❌ Do not restart services during the recording.
- ❌ Do not show `.env`, trunk credentials or the SSH key in any frame.

## VID-19 · AI pipeline deep dive

**Maps to:** original task 19 · **Est:** 4h · **Status:** ⬜ · **Script:** DOC-01 + DOC-05 + DOC-08

### Goal — done means
A recording walking through the voice pipeline code: STT → LLM → TTS. Where audio goes, how turns
work, how barge-in works, how silence detection works.

### Why it pairs with VID-14
VID-14 shows the pipeline running from the outside. This one opens it up. Recording them on the same
day means the same call is fresh in mind and can be referenced throughout.

### Plan
1. **Where audio goes** — walk the ingress chain from DOC-05 with the code open: RTP → µ-law decode →
   resample 8k→16k → STT. State the canonical rates and **why they are correct as they are** (the
   "misaligned sample rates" claim is false and was refuted with 655 production probes).
2. **How turns work** — `voice_pipeline/`: `turn_director` (orchestrates), `turn_runner` (executes),
   `turn_ender` (decides an utterance ended), `turn_streamer` (streams output), `turn_helpers`.
   Explain Deepgram Flux's turn-detection events: StartOfTurn, Update, EagerEndOfTurn, EndOfTurn,
   TurnResumed.
3. **How barge-in works** — `capture_mode`, the interrupt path, the utterance sequence counter, and
   content-aware echo immunity: why the agent must ignore *its own* audio without becoming deaf to the
   human. Explain why the blind 3-second grace window was rejected.
4. **Silence detection** — the silence monitor and its phrase ladder, and the history: it once died
   silently, which is a good story about why liveness matters.
5. **The LLM turn** — `llm_response`, `turn_streamer`, prompt assembly, `sentence_segmentation`
   splitting the stream into speakable units, and the compliance-floor plus per-model addendum prompt
   architecture.
6. **TTS out** — `tts_playback`, egress hygiene, resample 16k→8k, µ-law encode, RTP pacing.
7. **The other path** — `realtime_bridge` and the OpenAI Realtime topology, which is µ-law end to end
   and skips resampling entirely, plus the mid-call realtime→cascaded fallback.
8. **Resilience** — `resilient_stt`/`llm`/`tts`, the reconnect work from TKT-005, and provider
   concurrency caps.

### Do
- Reference moments from VID-14 by timestamp. Two videos that cross-reference are worth more than two
  that do not.
- Explain the *design tension* in barge-in: the agent must ignore its own echo but never the human.
  That framing makes the whole subsystem make sense.
- Cover the knowledge-base path (`knowledge_tool`, `kb_budget`) — it is on the turn path and affects
  latency.

### Don't
- ❌ Do not repeat the sample-rate misalignment claim.
- ❌ Do not conflate the cascaded and realtime paths.
- ❌ Do not skip silence detection because it is unglamorous. Dead air is the most common complaint
  about voice agents.
