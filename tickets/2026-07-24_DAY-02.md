## TKT-003 · Reconcile the two contradictory production deploy paths

**Maps to:** prerequisite for original tasks 10 and 15 · **Est:** 3h · **Status:** ⬜

### Goal — done means
One deploy mechanism is designated authoritative, and the other is either fixed to match reality or
made incapable of firing. `docs/v2/00-ground-truth.md` gains a "Deployment" section stating which is
which, and why.

### Why this is here
Two mechanisms in this repo claim to deploy production, and they describe different systems:

| | `deploy_to_server.sh` | `.github/workflows/deploy.yml` |
|---|---|---|
| Model | bare git checkout at `/opt/talky` | Docker image from GHCR |
| Apply | `git pull --ff-only` + `systemctl restart` | SCP `docker-compose.yml` + `docker compose up -d` |
| Trigger | manual | **automatic `workflow_run` after CI passes on `main`** |

`HANDOFF-NEXT-AGENT.md:25` states production is systemd, not compose. If that is right, then a
workflow that fires automatically on every green `main` build is either failing silently on every
push, or pointing at a host that is not production. Both are worth knowing before writing a runbook
(DOC-10) or recording a deployment video (VID-15) — see revision **R-2**.

**Depends on:** TKT-001. **Blocks:** DOC-10, VID-15, TKT-017.

### Plan
1. Read `.github/workflows/deploy.yml` end to end. Identify `DEPLOY_HOST`, `DEPLOY_PATH` and the
   trigger conditions.
2. Determine from the **GitHub Actions run history** whether the deploy workflow has ever succeeded,
   and when it last ran. `gh run list --workflow=deploy.yml` is enough.
3. Cross-check against TKT-001: is Docker even installed on the box?
4. Decide, and write the decision down with its reasoning:
   - **If systemd is authoritative** (expected): disable the automatic `workflow_run` trigger so it
     cannot fire, or gate it behind an explicit environment approval. Leave the file with a header
     comment explaining what it was and why it is dormant. Do not delete it — it is the
     starting point for a future containerised deploy.
   - **If the compose path is live somewhere** (staging?): document what it targets and how the two
     relate.
5. Confirm whether `deploy_to_server.sh:54`'s reference to `talky-voice-gateway` is correct given
   TKT-001's answer, and fix the script if it prints the status of a unit that does not exist.

### Do
- Check the actual Actions run history. The workflow's *intent* is in the file; its *behaviour* is in
  the run log.
- Preserve the dormant path with an explanatory comment. A future engineer finding a deleted deploy
  workflow in git history learns nothing; one finding a commented dormant file learns everything.
- Treat "the automatic trigger has been silently failing for weeks" as a finding for DOC-09, not an
  embarrassment to bury.

### Don't
- ❌ Do not test either path by deploying to production today.
- ❌ Do not delete `.github/workflows/deploy.yml`.
- ❌ Do not change `.github/workflows/ci.yml` gates — they are already blocking and green (**R-13**).
- ❌ Do not touch the server. This ticket is repo-side plus reading run history.

- [ ] Docker presence on prod confirmed from TKT-001

## TKT-004 · Live validation call + mine the unanalysed 2026-07-20 test calls

**Maps to:** validates original tasks 24, 28, 30, 32 · **Est:** 3h · **Status:** ⬜

### Goal — done means
One controlled outbound call is placed and analysed end-to-end, proving the greeting-truncation fix
works on real audio, plus a written analysis of the existing unanalysed test calls from
2026-07-20 ~11:25–11:30.

### Why this is here
Seven deploys landed over two days and **nothing shipped this week has been heard on a real call**.
The greeting fix (commit `33fee92c`) was justified by log evidence showing `interrupted=True` at ~1s
on both 07-20 test calls. That fix is unproven in production audio. Every doc and video in Sprints 2
and 3 describes a system nobody has recently listened to.

**Depends on:** TKT-001. **Blocks:** DOC-01, DOC-05, VID-14, VID-16 (which needs a real failed call
to trace).

### Plan
1. **Mine the existing calls first — it is free.** Pull logs for the 2026-07-20 11:25–11:30 window and
   analyse turn-taking, latency and greeting behaviour. This may answer the question without dialling.
2. Confirm the call setup with the user before dialling. Known-good configuration:
   - Tenant `info@allstateestimation.co.uk` (`790ca2db-6696-4fe9-9a2c-cd690c414a1e`)
   - Destination **`+442046132300`**, trunk `blaze-pool-150004`, caller-ID **`+442046132301`**
     (an earlier `+4420461323` was one digit short — verify before dialling)
   - LLM `qwen/qwen3.6-27b`; TTS currently **ElevenLabs** — switch to Cartesia in AI Options only if
     the user still wants that trial.
3. Place **one** call. Capture the full log window.
4. Verify against the checklist below and write `docs/v2/validation-call-2026-07-24.md`.

### Do
- Analyse the free evidence before generating new evidence.
- Watch Qwen 3.6 27B closely — it was pulled once before (2026-06-27) for dodging the AI-disclosure
  question and hallucinating. If it misbehaves, that is a finding, and a model rollback is a
  legitimate outcome of this ticket.
- Keep the recording. VID-14 and VID-16 both need real call material.

### Don't
- ❌ **Never start a campaign to test.** The allstate campaign holds ~691 real leads. One call, to one
  number the user names, placed deliberately.
- ❌ Do not dial without confirming the caller-ID digit count first.
- ❌ Do not restart services mid-call — a restart drops calls in progress.
- ❌ Do not change AI Options without telling the user which setting you changed and why.
