## VID-22 · Emergency procedures

**Maps to:** original task 22 · **Est:** 3h · **Status:** ⬜ · **Script:** DOC-11 + DOC-02 + DOC-10

### Goal — done means
A recording walking through four disaster scenarios with the actual commands: VPS down, database
problems, telephony stack crash, all calls failing simultaneously.

### Why it is recorded last
It draws on every other video. It is also the one most likely to be watched under pressure, so it must
be tight, ordered and free of digression.

### The core constraint, stated first on camera
**This is a single-VPS system.** One Hetzner box runs the API, all four workers, Asterisk and the
telephony edge. `talky-api` runs a single uvicorn worker because per-call ARI state is process-local.
There is no failover.

That is not a criticism — it is the reality every emergency procedure has to work within, and pretending
otherwise would make the procedures useless. Say it plainly at the start.

### Scenarios
Each: **detect → assess → act → verify → communicate.**

1. **VPS is down / unreachable**
   - Detect: external uptime monitor, total silence from the box
   - Assess: Hetzner console — is it network, or is the host down? Can you reach it out of band?
   - Act: console access, reboot from the panel, check disk and memory on the way back up
   - Verify: all units active, `/healthz/deep` 200, `/healthz/workers` 3/3, a validation call
   - Note: **calls in progress are lost.** There is no way around this. Say so.
2. **Database problems**
   - Detect: `/healthz/deep` failing on the DB check, connection errors in journald
   - Assess: connection exhaustion vs. slow queries vs. the DB itself being down. How to tell them apart.
   - Act: the safe actions only. **No `DROP`, no `TRUNCATE`, no row deletion, no schema change under
     pressure.** Restarting the API to release a leaked pool is legitimate; "cleaning up" data is not.
   - Verify: pool health, a diagnostic query, health probes
3. **Telephony stack crash**
   - Detect: calls failing at origination, `pjsip show registrations` empty, ARI errors
   - Assess: Asterisk process down vs. registration lost vs. carrier-side outage. Distinguishing these
     is the skill — a carrier outage looks identical from inside and is not fixed by restarting anything.
   - Act: restart Asterisk, confirm re-registration, then restart `talky-api` to re-establish ARI
   - Verify: registration state, then one validation call
4. **All calls failing simultaneously**
   - This is where the **stacked-blocker lesson** lands hardest. The documented incident had three
     independent causes at once: the call-guard abuse limiter, an IP-tier API 429, and an unregistered
     caller-ID. A campaign with 691 leads placed zero calls.
   - Detect / assess: check `call_guard_decisions`, provider quota status, trunk registration, and the
     dialer heartbeat — **all four**, not the first one that yields an answer
   - Act: address each cause found; **never disable a guard check to unblock a campaign**
   - Verify: one controlled call, then a small batch

Then:
5. **What must never be done in an emergency** — the standing list, on camera:
   no deletions of any kind · no `git reset --hard` / `git clean` / `git stash` on the server ·
   no DB writes without explicit authorisation · no restart during live traffic without checking ·
   never disable a call guard · never flip `CLEANUP_DRY_RUN` under pressure.
6. **Rollback** — the fastest safe path back to a known-good commit, per DOC-10, including restoring
   systemd unit files if the bad deploy changed them.
7. **Communication** — who to tell, what to say, and where the incident is written down.

### Do
- Rehearse the whole thing before recording. This one must be tight.
- Say what is *not* recoverable — in-progress calls are lost on any restart. False hope wastes minutes.
- Show the "never do this" list on screen, not just in narration. It is the part that must survive
  under stress.

### Don't
- ❌ **Do not actually cause any of these emergencies.** Walk the commands; explain what each does;
  execute only the genuinely safe read-only ones.
- ❌ Do not demonstrate a VPS reboot on production.
- ❌ Do not show the Hetzner console credentials or the SSH key.
- ❌ Do not present a procedure that has not been thought through end to end. A wrong emergency
  procedure is worse than no procedure.

- [ ] Stacked-blocker lesson delivered in scenario 4

   tenant filters, realtime fallback, circular-import AST test, CI gates, guard caching, cleanup timer,
   prod-gate Stripe checks, greeting echo immunity.
