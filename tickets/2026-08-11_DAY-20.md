## DOC-10 · Deployment runbook

**Maps to:** original task 10 · **Est:** 4h · **Status:** ⬜ · **Depends on:** TKT-003

### Goal — done means
`docs/v2/10-deployment.md`: step by step how to deploy a change, how CI works, how to roll back, how
to verify a deployment succeeded, what to do when it fails.

### Why TKT-003 is a hard dependency
Until Day 02 there were **two contradictory deploy mechanisms** in this repo — a systemd/git path and
a Docker-Compose/GHCR GitHub Actions path that fired automatically on every green `main` build. A
runbook documenting the wrong one is worse than no runbook. See revision **R-2**. Write this from
TKT-003's designated authoritative path and say explicitly what the other one is and why it is dormant.

### Plan
1. **CI** — `.github/workflows/ci.yml`, job by job, and what each blocks on:
   `secret-scan` (gitleaks) → `secret-guard` → `backend` (ruff · pip-audit with one reviewed CVE
   exception · bandit `-lll` · schema init · Alembic upgrade/downgrade/upgrade round-trip · pytest
   with `--cov-fail-under=20` · OpenAPI export) · `frontend` (npm audit high · lint · `tsc --noEmit` ·
   build) · `docker-build` (Trivy, fails on fixed CRITICAL/HIGH) · `schema-check`.
   Note that the root `ci.yml` was a decoy that GitHub never read, and was deleted in TKT-013.
2. **Pre-deploy gate** — the full test suite at or above baseline with **zero collection errors**.
   Record the actual expected number. If `--continue-on-collection-errors` is ever needed, something
   regressed and the deploy stops.
3. **The deploy sequence**, exactly, as verified by seven successful production deploys:
   ```bash
   git push origin main
   # server
   cd /opt/talky && git pull --ff-only origin main
   cd backend && venv/bin/python -c 'import app.main; print("IMPORT_OK")'
   sudo systemctl restart talky-api talky-dialer-worker talky-voice-worker talky-reminder-worker
   ```
4. **systemd unit changes need the full ritual** and it must be a separate, prominent section: edit the
   repo copy → `cp` to `/etc/systemd/system/` → `daemon-reload` → restart → **verify it actually came
   up** (`systemctl is-active` plus `systemctl show <unit> -p Type,Restart,WatchdogUSec`). The three
   worker units are `Type=notify`; a worker that fails to send `READY=1` leaves systemd hanging in
   `activating` and then failing the unit. This has bitten before.
5. **Verification** — `/healthz/deep` 200 (retry: a `000` immediately after restart is a pre-bind race,
   not a failure), `/healthz/workers` 3/3, `journalctl -p err` clean. Note that `/openapi.json` is
   disabled in production and a global auth middleware returns **403 for every unauthenticated request
   before method matching**, so route existence cannot be probed unauthenticated — prove routes via a
   clean import plus restart instead.
6. **Rollback** — `git checkout <hash>` on the server plus restart; and if systemd units changed in the
   bad deploy, restore the previous unit files and `daemon-reload` too. That second half is the part
   people forget.
7. **When it fails** — import error, unit stuck in `activating`, health probe failing, migration
   trouble. One short procedure each.
8. Frontend: Vercel auto-deploys from `main`. Document how to verify a frontend deploy and how to roll
   one back.

### Do
- Have the reviewer perform a real deployment following only this doc (that is VID-15 on Day 27 — the
  video is the verification).
- Include the "check for active calls before restarting" step. A restart drops calls in progress.
- Write the failure procedures for someone under pressure: short, ordered, no prose.

### Don't
- ❌ Do not document the dormant compose path as live.
- ❌ Do not omit the systemd ritual. It is the step most likely to be skipped and most likely to hurt.
- ❌ Do not include the sudo password. It is deliberately in no repo file — the doc says "ask the user".

## DOC-11 · Incident runbook

**Maps to:** original task 11 · **Est:** 4h · **Status:** ⬜ · **Depends on:** DOC-01, DOC-02, DOC-05

### Goal — done means
`docs/v2/11-incidents.md`: how to diagnose and fix the five most common production issues —
degraded status, calls failing, dialer stopped, audio distortion, API errors.

### Plan
Each of the five gets the same structure: **Symptom → First checks → Likely causes, ordered by
probability → Fix → Verify → Escalate.** No prose paragraphs; this is read at 3am.

1. **Degraded status** — `/healthz/deep` non-200 or `/healthz/workers` reporting unhealthy. Which
   dependency, how to confirm, what a single unhealthy worker means versus all three. Note the known
   good-behaviour case: the reminder worker does not retry Redis at boot, so it correctly reports
   unhealthy while DB work continues — right fail-direction, but it will alarm someone.
2. **Calls failing** — split the diagnosis by stage, because the symptom is identical and the causes
   are not: guard-blocked · origination failure · no answer · media failure. **Include the documented
   real incident**: three stacked blockers (call-guard abuse limiter, IP-tier API 429, unregistered
   caller-ID) made a 691-lead campaign place **zero** calls. That is the worked example — one symptom,
   three independent causes, none visible without looking in three places.
3. **Dialer stopped** — heartbeat key in Redis, `/healthz/workers`, the systemd watchdog. Note the
   history: the heartbeat coroutine existed but was **never scheduled**, so the worker could die
   invisibly. That is why three layers of liveness now exist. Also cover campaign schedule gating
   (`ignore_schedule`, timezone, `call_gap_seconds`, `batch_size`) — a campaign that looks stopped is
   often correctly waiting.
4. **Audio distortion** — link straight to DOC-05's symptom→stage table rather than duplicating it.
   Cover the two known-and-fixed buzz sources (near-zero µ-law egress; event-loop-blocking DB teardown
   causing underrun) so a recurrence is recognised instantly.
5. **API errors** — auth 403s (remember: global auth middleware returns 403 before method matching),
   rate limiting, DB pool exhaustion, provider 429s. How to tell them apart from a log line.

Then:
6. **Log triage** — the `journalctl` invocations that are actually useful, per unit, with the grep
   patterns worth knowing.
7. **A decision tree at the top.** "Calls are failing" is where most incidents start; the tree routes
   from there to the right section in one hop.
8. **Escalation and safety.** What must never be done during an incident: no deletions, no
   `git reset --hard` / `git clean` / `git stash` on the server, no DB writes without explicit
   authorisation, no restart during live traffic without checking, never disable a guard to "unblock"
   a campaign.

### Do
- Ground every section in a real incident where one exists. This project has good material — use it.
- Put the safety list in the runbook itself, not in a linked document. Under pressure, nobody follows
  the link.
- Order causes by actual probability, not by how interesting they are.

### Don't
- ❌ Do not write procedures that delete anything.
- ❌ Do not recommend disabling call-guard checks as an incident remedy. That is how an illegal call
  gets placed during an outage.
- ❌ Do not duplicate DOC-05's audio table — link it. Two copies diverge.
