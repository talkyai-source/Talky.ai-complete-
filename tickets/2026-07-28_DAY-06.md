## TKT-007 · Flux concurrency guard (F-07)

**Maps to:** batch 2 increment #10 · **Est:** 2h · **Status:** ⬜

### Goal — done means
The Flux STT adapter respects the provider concurrency cap that Nova already respects.

### Why this is here
`deepgram_nova.py` honours the per-provider concurrency limiter; `deepgram_flux.py` does not. Under
concurrent call load, Flux opens sockets past the cap, and the failure mode is provider-side
throttling that arrives as apparently-random STT failures across *all* calls — the worst kind of
production symptom, because it presents as many small unrelated bugs.

**Depends on:** TKT-006 (same files — do not stack unshipped changes on them).
**Blocks:** nothing.

### Plan
1. Read how `deepgram_nova.py` acquires from `infrastructure/providers/provider_concurrency.py`.
2. Apply the same acquisition to Flux. Same limiter, same key — **do not invent a second mechanism**.
3. Decide and document the over-cap behaviour: queue with timeout, or fail over to the secondary
   provider. Failing the call outright is not acceptable.
4. Test under simulated concurrent sessions.

### Do
- Reuse the existing limiter exactly. Two concurrency mechanisms against one provider quota is worse
  than one.
- Make the over-cap path observable — a log line at minimum. A silent cap is indistinguishable from
  a hang.

### Don't
- ❌ Do not raise the cap to make the problem go away. The cap reflects the provider contract.
- ❌ Do not hold the limiter slot across an entire call if Nova does not. Match the existing lifetime.

### Checklist
- [ ] Flux acquires from the shared limiter
- [ ] Over-cap behaviour decided, implemented, documented
- [ ] Over-cap path emits a log line
- [ ] Concurrency test added
