## DOC-12 · Third-party services and accounts

**Maps to:** original task 12 · **Est:** 3h · **Status:** ⬜

### Goal — done means
`docs/v2/12-third-party.md`: every external service — what it is for, which account owns it, where
credentials are stored, what the billing situation is.

### Why this one is quietly critical
This is the **bus-factor document**. If one person holds the vendor logins, the company's ability to
operate is one unavailable person wide. It is also the doc most likely to be skipped because it feels
administrative rather than technical.

### Plan
1. Verify each row against the code — that the adapter exists and is wired into its factory.
2. For each: **who owns the account**, where credentials live (vendor dashboard / server `.env` /
   `secrets/` / GitHub secrets / Vercel env), billing model, and what happens when it is exhausted or
   the card fails.
3. **Flag the single points of failure explicitly.** The SIP carrier is the sharpest: if Blazedigitel
   is down, the product does not work, and Twilio failover exists but is not proven under real traffic.
4. ⚠️ **Record the hardcoded carrier credentials in `setup-asterisk.sh` as a security finding** —
   reference the file, never reproduce the secret. Rotating them and moving them into `secrets/` is a
   concrete follow-up item for REV-40.
5. Note which services have a **free/trial tier** that will expire, and which have no spend alerting.
   A provider quota exhausting mid-campaign presents as "random call failures".
6. Cross-link DOC-06 for the variable names and DOC-08 for the provider behaviour.

### Do
- Name a human owner per account. "The team" is not an owner.
- Record renewal and expiry dates where they exist.
- Say plainly which accounts have no second person with access. That is the finding.

### Don't
- ❌ **Never write a credential, key or account password into this doc.** Location and owner only.
- ❌ Do not commit this doc to a public remote without review — it is an infrastructure map.
- ❌ Do not list a provider as "in use" without verifying its adapter is actually reachable in code.

## DOC-09 · Known issues and technical debt

**Maps to:** original task 9 · **Est:** 4h · **Status:** ⬜ · **Depends on:** DOC-01 … DOC-12 and all of Sprint 1

### Goal — done means
`docs/v2/09-known-issues.md`: every part of the codebase that is fragile, temporarily patched, or
likely to break under load. **Honest assessment. No sugarcoating** — the original document's words,
and the right standard.

### Why it is last
It synthesises the other eleven docs plus every Sprint 1 finding. Written first, it would be written
from memory. Written last, it is written from evidence.

### Seed content — already known, must appear
From this planning exercise and Sprint 1. Verify each before publishing; do not copy blind.

**Dead / contradictory code**
- `post_call_analyzer.py` — fully implemented, referenced only by its own test. Delete or wire in.
- `docs/OIDC_INTEGRATION.md` — 130 lines documenting an integration with no implementation found under
  `backend/app`.
- Committed C++ build artifacts (`build/`, `build-asan/`) in the tree.

**Architecture / structure**
- Configuration is decentralised: ~37 typed settings versus **245** env vars read via `os.getenv`.
- A 4-class ORM (`storage/models.py`) sits beside a ~70-table schema; most access is raw SQL.
- Two migration mechanisms: Alembic plus 47 raw SQL files.
- `talky-api` must run `--workers 1` because per-call ARI state is process-local. **This is the
  scaling ceiling** and it is a v2 architectural question, not a config tweak.
- Three separate, non-unified monitoring configuration sets, none confirmed scraping production.
- 552 markdown files; `docs/v2/` is now the canonical set and the rest is history.

**Security**
- Hardcoded carrier credentials in `setup-asterisk.sh`.
- If TKT-001 found `talky-voice-gateway` running from an unversioned unit file — record it.
- Bandit backlog: 60 Medium / 214 Low. High is 0 and gated.

**Operational**
- `CLEANUP_DRY_RUN=true` — retention is **not** actually enforced yet.
- The batch-1 migration and any other unapplied migration, until TKT-013 closes it.
- Whatever TKT-009's audit found; whatever TKT-003 found about the dormant deploy workflow.
- No load-test evidence for the concurrency ceiling — nobody knows where the knee is.

**Deferred by decision (not oversight — record the reasoning)**
- Sample-rate "alignment" — refuted with production evidence.
- Blind 3-second barge-in grace — would suppress real interrupts.
- DNC caching — compliance. A newly-added DNC number must never be dialable from a stale cache.
- PgBouncer, if TKT-009's verdict was DEFER.
- Twilio media session resumption — not possible; the socket terminates in-process.

### Plan
1. Read every Sprint 1 ticket and every Sprint 2 doc; harvest what the running findings file missed.
2. Structure by category (above), and within each rank by **risk × likelihood**, not by how easy it
   is to fix.
3. For each item: what it is, why it is that way, what it would take to fix, and what happens if it is
   not fixed. That fourth field is what makes this a decision document rather than a complaint list.
4. Mark each **v1-acceptable** or **v2-blocking**. That distinction is the entire input to REV-40 and
   REV-42.
5. Add a "load and scale" section stating honestly what is unknown. `talky-api` at one worker and no
   load-test data is the most important unknown in the system.

### Do
- Be genuinely honest. A sanitised version of this document is worse than none, because it will be
  trusted.
- Record *why* things are as they are. Most were deliberate trade-offs under time pressure, and the
  reasoning is what a future engineer needs in order to change them safely.
- Include the deliberate rejections. "We considered this and rejected it because X" prevents the same
  bad idea being re-proposed every quarter — and it has already been re-proposed once here.

### Don't
- ❌ Do not soften. The task says no sugarcoating.
- ❌ Do not list an item without a "what happens if we don't fix it".
- ❌ Do not fix things while writing this. Record them. Fixes are v2 tickets or freeze exceptions.
