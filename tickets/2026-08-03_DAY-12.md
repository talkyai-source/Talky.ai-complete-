## Shared rules for TKT-014 / TKT-015 / TKT-016

These three tickets are the same operation on three files. The rules below apply to all of them and
are not repeated in each.

**The split is mechanical, not creative.** Move code; do not rewrite it. If you find a bug while
splitting, write it down for DOC-09 and fix it in a **separate commit** — never inside the move. A
diff that both moves and changes code is a diff nobody can review, on the live call path.

**Do**
- One file per commit. Each independently revertible.
- Split along responsibility seams that already exist in the file, not at arbitrary line counts.
  "Under 500 lines" is the goal, not the method — six coherent modules beat five arbitrary ones.
- Keep the public entry points where callers already import them from, re-exporting if needed. Every
  call site you do not have to touch is a call site you cannot break.
- Run the full gate after **each** file, not once at the end.
- Confirm `tests/unit/test_no_domain_api_imports.py` still passes. It forbids the domain layer
  importing from `app.api`, with two allowlisted pre-existing residuals (`campaign_service.py`,
  `state_backend.py`). A split is an easy way to accidentally introduce a third.

**Don't**
- ❌ Do not change behaviour. Not even "obviously better" behaviour.
- ❌ Do not split all three in one commit.
- ❌ Do not add the split modules to the AST test allowlist to make it pass. If it fails, the split
  introduced a layering violation — fix the split.
- ❌ Do not deploy a split without a validation call afterwards. Unit tests do not hear dead air.
- ❌ Do not `git stash` in a shared tree.
