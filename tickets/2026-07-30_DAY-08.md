## TKT-010 · PgBouncer connection pooling (fix 13) — **conditional**

**Maps to:** original task 29 · **Est:** 4h · **Status:** ⬜ · **Gate:** TKT-009 verdict = PROCEED

### ⛔ Entry gate
**Do not open this ticket if TKT-009's verdict is DEFER.** If deferred: mark this ticket ⬛ with the
reason, record PgBouncer as a v2 item in DOC-09 and REV-40, and spend the day on TKT-011 plus pulling
TKT-012 forward. Shipping connection pooling over an unresolved RLS-bypass leak would trade a
performance win for a cross-tenant data-exposure risk. That is not a trade this project makes.

### Goal — done means
The backend connects through PgBouncer, tenant isolation is proven intact under pooling, and the
rollback path is a one-line configuration revert.

### Why this is here
Direct Postgres connections cap concurrency and make connection-storm recovery slow. `infra/pgbouncer/pgbouncer.ini`
already exists in the repo. The only thing that ever blocked this was the RLS hazard TKT-009 clears.

Note the document's version of this fix says "pgbouncer in docker-compose". **Production is systemd,
not compose** (revision R-2) — PgBouncer is a host service here, installed and unit-managed like
everything else on the box.

**Depends on:** TKT-009 (hard gate), TKT-003 (deploy model). **Blocks:** nothing.

### Plan
1. Review `infra/pgbouncer/pgbouncer.ini`. Confirm pool mode. **Transaction** mode is the goal and
   the reason TKT-009 exists; session mode would sidestep the hazard but forfeit most of the benefit.
2. `backend/app/core/db.py`: point at port 6432 and set **`statement_cache_size=0`** — asyncpg's
   prepared-statement cache is incompatible with transaction pooling and this is the classic way
   this integration fails in production, days later, under load.
3. Install PgBouncer as a host service with its own systemd unit. Unit file goes **in the repo**
   (`backend/systemd/`) — an unversioned unit is the risk already flagged in R-4.
4. Stage locally first. Run the full suite against a local PgBouncer, not just against Postgres.
5. Deploy behind an environment switch so rollback is a config change plus a restart, not a revert.

### Do
- Test the **tenant-isolation suite** specifically against pooled connections. That is the risk this
  whole two-ticket sequence exists to manage.
- Keep the switch. First-day-of-pooling problems appear under real concurrency, which local tests do
  not reproduce.
- Watch for `prepared statement "__asyncpg_stmt_x__" already exists` after deploy — that is the
  `statement_cache_size` symptom, and it appears intermittently rather than immediately.

### Don't
- ❌ Do not proceed without the PROCEED verdict.
- ❌ Do not skip `statement_cache_size=0`.
- ❌ Do not add PgBouncer to `docker-compose.yml` and call it deployed — prod does not read that file.
- ❌ Do not deploy this on a Friday. It is here on a Thursday on purpose.

## TKT-011 · Twilio + Vonage session registry (fix 15)

**Maps to:** original task 30 (second half) · **Est:** 3h · **Status:** ⬜

### Goal — done means
In-process telephony session dictionaries are replaced by a registry with a defined
restart-recovery behaviour, and no orphaned `calls` row is left `in-progress` forever after a restart.

### Why this is here
The V2 document proposes moving `_twilio_sessions` to an external state backend "so sessions survive
a restart". **They cannot.** Twilio's media WebSocket terminates in this process; when the process
dies the socket dies with it, and no external channel exists to reconnect through. This differs
fundamentally from Asterisk/ARI, where the call still exists in Asterisk and can be re-attached.
See revision **R-12**.

What *is* achievable, and what actually matters, is that a restart does not leave the database
claiming calls are still running.

`vonage_bridge._vonage_sessions` has the same class of bug and is **worse**: its `/event` handler
reads the dict across requests, so a restart between the call starting and its event arriving loses
the correlation entirely. Both are in scope.

**Depends on:** TKT-001 (is Twilio/Vonage traffic real today?). **Blocks:** nothing.

### Plan
1. Build `TwilioSessionRegistry` — an explicit object with a clear lifecycle, replacing the bare dict.
2. Give it a startup recovery pass: find `calls` rows this process previously owned that are still
   marked in-progress, and mark them **ended with a reason** (`process_restart`). Not "recovered" —
   honestly ended.
3. Apply the same treatment to `_vonage_sessions`, including the cross-request `/event` read.
4. Document explicitly, in code and in DOC-01, that **media sessions do not survive restart** and why.
   Someone will otherwise try to make them survive again in v2.

### Do
- Make the recovery reason visible in the data. An operator seeing `ended: process_restart` learns
  something; one seeing `ended` learns nothing.
- If TKT-001 shows Twilio/Vonage carry no real traffic, still fix it — but say so in the ticket, and
  let that inform how much time it deserves versus TKT-012.

### Don't
- ❌ Do not implement session resumption for Twilio media. It is not possible; building it would
  produce a recovery path that silently never works.
- ❌ Do not delete orphaned `calls` rows. Mark them ended. **Nothing on this project deletes rows.**
- ❌ Do not leave the Vonage bug unfixed because Twilio was the one named in the document. It is the
  worse of the two.

- [ ] Startup recovery marks orphaned rows ended with reason `process_restart`
- [ ] Vonage `_vonage_sessions` given the same treatment
- [ ] Vonage cross-request `/event` read fixed
- [ ] "Media sessions do not survive restart" documented in code + noted for DOC-01
- [ ] Tests for the recovery pass
