## TKT-008 · Nova normalization + config parity — including the email-truncation bug

**Maps to:** batch 2 increment #11 · **Est:** 3h · **Status:** ⬜

### Goal — done means
Flux and Nova produce equivalently-shaped transcript output under equivalent configuration, and the
**trailing-fragment truncation bug is fixed** — the one that drops the end of an email address.

### Why this is here
`turn_text()` returns the finalized transcript segments **or** the interim tail — not both. When a
caller says "…dot com" and the turn ends before that fragment finalizes, the fragment is discarded.
The user's address becomes `john@example` and the lead is worthless.

Email addresses and phone numbers are **CORE fields on this project**: they must be exactly right,
and correctness beats latency every time. This is not a transcript-quality nicety, it is a
data-integrity defect on the field the whole call exists to capture.

**Depends on:** TKT-006. **Blocks:** nothing.

### Plan
1. Reproduce first. Write a failing test that feeds finals + a trailing non-final fragment and
   asserts the fragment survives. Do not fix anything until that test is red for the right reason.
2. Fix `turn_text()` to compose finals **and** the interim tail, with de-duplication where the tail
   overlaps the last final. Overlap is the trap: naive concatenation yields `example.comcom`.
3. Audit config parity between Flux and Nova — sample rate, encoding, endpointing, language,
   keyterms. Record every divergence and whether it is deliberate.
4. Normalize Nova's output shape to Flux's `TranscriptChunk` contract so downstream turn logic does
   not need to know which provider produced a chunk.

### Do
- Test with real-world shapes: `john.smith@example.co.uk`, `+44 7700 900123`, spoken digit
  sequences, spelled-out letters, `"at"` vs `"@"`.
- Treat every Flux/Nova config divergence as deliberate-or-a-bug. Write down which.

### Don't
- ❌ Do not concatenate finals and the tail without overlap handling.
- ❌ Do not "fix" this by extending the turn timeout — that trades a data bug for a latency bug and
  still loses the fragment when the caller stops talking.
- ❌ Do not build a brittle hand-written email parser. The established approach here is deterministic
  handling of the `@`-syntax only, the LLM assembling the rest, and the confirmation loop as the
  guarantee. Keep it.

### Checklist
- [ ] Failing test written and red before any fix
- [ ] `turn_text()` composes finals + tail with overlap de-duplication
- [ ] Tested against emails, phone numbers, spelled sequences
- [ ] Flux/Nova config divergences enumerated and classified
- [ ] Nova output normalized to the shared chunk contract
- [ ] Gate green
- [ ] Peer-reviewed

### Test cases

|---|---|---|---|
| 1 | Finals + trailing non-final `"dot com"` | full address preserved | ⬜ |
| 2 | Tail overlapping the last final | de-duplicated, not doubled | ⬜ |

## TKT-009 · RLS `SET` → `SET LOCAL` audit — gate for PgBouncer

**Maps to:** prerequisite for original task 29 · **Est:** 3h · **Status:** ⬜

### Goal — done means
Every `SET` of an RLS-related Postgres session variable in the codebase is inventoried and classified
as safe-under-transaction-pooling or not, with a written verdict on whether fix 13 (PgBouncer) can
proceed.

### Why this is here
`backend/app/workers/dialer_worker.py` sets **`app.bypass_rls = 'on'` at session level**, with a
comment relying on it persisting across statements on a pooled connection. Under PgBouncer
*transaction* pooling that assumption is void — and the state that would leak is **RLS bypass**, on a
multi-tenant database. The realistic worst case is one tenant's connection inheriting bypass and
reading another tenant's data.

That is why fix 13 is not a configuration change. See revision **R-11**.

**Depends on:** nothing. **Blocks:** TKT-010 — hard gate.

### Plan
1. Inventory every occurrence:
   ```bash
   cd backend && grep -rn "SET app\.\|SET LOCAL app\.\|set_config" app/ --include=*.py
   ```
2. For each: file, function, variable, session-or-local, and whether it is inside an explicit
   transaction. Build the table in `docs/v2/rls-set-audit.md`.
3. Classify: **SAFE** (`SET LOCAL` inside `async with conn.transaction()`) / **UNSAFE** (session-level
   on a pooled connection) / **N/A** (dedicated non-pooled connection — must be proven, not assumed).
4. Convert every UNSAFE site to the pattern the adapter layer already uses correctly:
   ```python
   async with conn.transaction():
       await conn.execute("SET LOCAL app.bypass_rls = 'on'")
       ...
   ```
   Note this pattern was already applied once this project, to fix `cleanup_worker._count_expired`,
   which was silently skipping rows forever because it lacked the bypass.
5. Write the verdict: **can fix 13 proceed, yes or no.**

### Do
- Include `set_config(...)` calls — they are the same mechanism by another name and grep for `SET`
  alone will miss them.
- Prove any "this connection is never pooled" claim by reading how the connection is obtained.
  An assumption is what created this hazard.
- Add a regression test that fails if a session-level RLS `SET` is reintroduced. The AST
  architecture test in `tests/unit/test_no_domain_api_imports.py` is the working precedent for
  locking an invariant permanently.

### Don't
- ❌ Do not start TKT-010 (PgBouncer) if any site remains UNSAFE.
- ❌ Do not remove the dialer's bypass because it looks dangerous — it is there for a reason. Change
  its **scope**, not its existence.
- ❌ Do not test RLS behaviour by writing to production. Read-only, or local.

- [ ] Each classified SAFE / UNSAFE / N/A with reasoning
- [ ] `docs/v2/rls-set-audit.md` written
- [ ] All UNSAFE sites converted to `SET LOCAL` in an explicit transaction
- [ ] Regression test added that forbids reintroduction
- [ ] Tenant-isolation tests still green (`tests/security/test_idor_tenant_scoping.py`)
- [ ] **Written verdict on fix 13: PROCEED / DEFER**
