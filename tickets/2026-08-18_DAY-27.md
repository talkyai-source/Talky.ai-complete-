## VID-15 · End-to-end deployment

**Maps to:** original task 15 · **Est:** 3h · **Status:** ⬜ · **Script:** DOC-10

### Goal — done means
A recording of a **real deployment**: make a small code change, push to GitHub, watch CI run, deploy
to the VPS, verify it is live. First keystroke to production.

### Why it must be real
This recording is also **DOC-10's verification**. If the runbook is followed on camera and it works,
the runbook is proven. If a step is missing, the video finds it. That is why VID-15 comes after
DOC-10 rather than alongside it.

### Plan
1. Pick a genuinely trivial, genuinely real change — a documentation typo, a log-message clarification.
   Something with zero behavioural risk that is nonetheless worth shipping.
2. Record from the first keystroke:
   - make the change; run the **pre-deploy gate** (`pytest tests/unit tests/security -q`) and state the
     expected number out loud before it finishes
   - commit with a message that explains *why*, not what
   - push; open GitHub Actions and **watch CI run**, narrating each job and what it blocks:
     secret-scan → secret-guard → backend (ruff, pip-audit, bandit, schema, Alembic round-trip, pytest,
     OpenAPI) → frontend → docker-build → schema-check
   - **explain that CI passing does not deploy anything** — deployment is the deliberate manual step
     below. This is the single most common misconception about this repo, and TKT-003 exists because
     the repo itself used to be ambiguous about it. Explain what the dormant compose workflow is and
     why it is dormant.
   - deploy per DOC-10: `git pull --ff-only` → import smoke test → `systemctl restart`
   - **check for active calls before restarting**, on camera
   - verify: `/healthz/deep` (mention the `000` pre-bind race and retry), `/healthz/workers` 3/3,
     `journalctl -p err`
3. **Demonstrate a rollback** — identify the previous good commit, explain the procedure including
   restoring unit files if units changed, and either perform it and re-deploy forward, or walk it
   step by step without executing. Say which you are doing and why.
4. Show the frontend side: Vercel auto-deploys from `main`; how to confirm.

### Do
- Say the expected test count before the gate finishes. That is what treating a number as a gate looks
  like, rather than glancing at green.
- Show the active-call check. It is the step most likely to be skipped and the one that drops real calls.

### Don't
- ❌ Do not deploy anything behaviourally risky for a recording.
- ❌ Do not skip the gate to save recording time. If the gate is slow, that is honest.
- ❌ Do not deploy while a real call is in progress. If one is, wait — and narrate the wait. That is
  the lesson.
- ❌ Do not show the sudo password or the SSH key.

- [ ] CI jobs narrated with what each blocks
- [ ] "CI green ≠ deployed" stated explicitly; dormant workflow explained
- [ ] Active-call check performed on camera
- [ ] Deploy + verification shown
- [ ] Rollback demonstrated or walked step by step, incl. unit-file restoration
- [ ] Vercel side covered

## VID-16 · Debugging a failed call

**Maps to:** original task 16 · **Est:** 4h · **Status:** ⬜ · **Script:** DOC-11 + DOC-01

### Goal — done means
A recording taking a **real failed call** from the logs and diagnosing it live: where to look, what
the errors mean, how to get from symptom to root cause.

### Sourcing the failure — in order of preference
1. **A genuine recent production failure.** Best material by far. Check the logs from the last two
   weeks; if the opportunistic capture on Day 24 caught one, use it.
2. **The documented historical incident**: three stacked blockers — the call-guard abuse limiter, an
   IP-tier API 429, and an unregistered caller-ID — made a 691-lead campaign place **zero** calls.
   This is exceptional teaching material: one symptom, three independent causes, none visible without
   looking in three separate places. It defeats the instinct to stop at the first cause found.
3. **A safely induced failure** — e.g. dial an invalid destination. Last resort, and say on camera
   that it is induced.

### Plan
1. Start from the **symptom only**, as a real report would arrive: "the campaign isn't calling anyone"
   or "the call dropped". Do not reveal the cause.
2. Walk DOC-11's decision tree on camera, out loud, including the branches you rule out and *why* —
   the ruling-out is the skill.
3. Show the log locations for each stage: dialer journal, API journal, voice worker journal, and the
   database (`calls`, `dialer_jobs`, `call_guard_decisions`).
4. Interpret real error lines. What each actually means versus what it appears to mean.
5. Reach root cause. Then — the important part — **ask whether there is another cause behind it**. The
   691-lead incident is the case study for exactly this.
6. Show the fix, or explain it if applying it live is unsafe.
7. Close with the general method: symptom → stage → evidence → cause → verify.

### Do
- Keep the dead ends in. A debugging video where every guess is right teaches nothing about debugging.
- Show what a *healthy* log looks like first, briefly. You cannot recognise wrong without knowing right.
- Use the real incident. It is better than anything that could be manufactured.

### Don't
- ❌ Do not induce a failure by breaking production. If inducing, use an invalid destination, never a
  disabled service or a broken config.
- ❌ Do not disable a call-guard check to demonstrate anything. That is how an illegal call happens.
- ❌ Do not show customer data beyond what is agreed for VID-18.
- ❌ Do not present a tidied-up narrative. Real diagnosis is untidy and the untidiness is instructive.

### Checklist
- [ ] Failure sourced (real / historical / induced — stated on camera)
- [ ] Starts from symptom only
- [ ] DOC-11 decision tree walked, incl. ruled-out branches
- [ ] Log locations shown for each stage
- [ ] Real error lines interpreted
- [ ] Root cause reached
- [ ] "Is there another cause behind it?" demonstrated
- [ ] Fix shown or explained
- [ ] General method summarised at the close
- [ ] ≤60 min · saved to shared drive · reviewed by: ______
