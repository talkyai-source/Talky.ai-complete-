## Standing rules for every Sprint 2 doc ticket

Applies to DOC-01 … DOC-12; not repeated per ticket.

1. **Destination is `docs/v2/`.** That directory is the canonical set. There are 552 markdown files
   in this repo and not one is authoritative — adding a 553rd loose file solves nothing.
2. **Supersede, don't accumulate.** Any existing doc your new one replaces gets a header line at the
   top: `> ⚠️ SUPERSEDED by docs/v2/<file>. Kept for history.` Known drifted docs are listed in
   `00-PLAN-REVISIONS.md`.
3. **No line numbers.** Reference modules and functions. Line numbers rot; the Day 12–13 splits just
   proved it.
4. **Write for someone who joined yesterday.** The document's stated bar: *any engineer can understand
   what Talky.ai is, how it works, and where everything lives — without asking anyone.* If a sentence
   only makes sense to someone who already knows, rewrite it.
5. **Every doc opens with a "Last verified" line** — date + the commit it was verified against.
6. **Peer review before 🟢.** Ground rule 5. The reviewer's name goes in the ticket.
7. **Anything you discover that is wrong, fragile or surprising goes into the DOC-09 running findings
   file the moment you find it.** Writing documentation is the best bug-finding exercise this project
   will run all month. Do not let those findings evaporate.

## DOC-01 · Full call flow document

**Maps to:** original task 1 · **Est:** 4h · **Status:** ⬜

### Goal — done means
`docs/v2/01-call-flow.md`: every step from `dialer_worker` picking up a job to the post-call summary
being written. Every service, file and function named. With a diagram.

### Why this is first
It is the spine. DOC-05, DOC-08, DOC-11, VID-14, VID-16 and VID-19 all reference it. Get it right and
they get easier; get it wrong and six deliverables inherit the error.

### ⚠️ Correction the original task requires
The V2 document says the flow ends with **`post_call_analyzer`** running after hangup. It does not.
`backend/app/domain/services/post_call_analyzer.py` is fully implemented and referenced **only by its
own unit test** — it is dead code. The live chain is:

```
workers/dialer_worker.py : DialerWorker._run_loop
  → process_job → _evaluate_call_guard → _make_call
    → POST /api/v1/sip/telephony/call
      → api/v1/endpoints/telephony_bridge.py : make_call
        → CallGuard (DNC, rate, concurrency, geo, hours, subscription)
        → infrastructure/telephony/asterisk_adapter.py : originate_call  (ARI)
[Asterisk events return through the adapter callback registry]
  → domain/services/telephony/lifecycle.py : _on_early_ringing / _on_ringing
      → voice_orchestrator : create_voice_session
  → lifecycle.py : _on_new_call → pipeline.start_pipeline
      → domain/services/voice_pipeline/  (turn_runner, turn_director, turn_streamer, …)
  → lifecycle.py : _on_audio_received     (caller audio, per frame)
[hangup]
  → lifecycle.py : _on_call_ended
      → call_status.record_call_state_by_provider_id
      → global_concurrency.release_lease
      → services/scripts/call_transcript_persister.py : save_call_transcript_on_hangup
          → _schedule_call_summary
            → call_summary/store.py : generate_and_store
              → call_summary/summarizer.py : summarize_transcript   ← the real post-call analyzer
```

**Verify this chain against the post-split code before writing it down.** Module paths under
`lifecycle.py` and `voice_orchestrator.py` changed on Days 12–13.

### Plan
1. Re-trace the chain against post-freeze code. Correct the map above where the splits moved things.
2. Draw the diagram (Mermaid, so it lives in the markdown and stays reviewable in a diff).
3. Document each stage: what it does, what it needs, what happens when it fails, where the log lines
   are.
4. Document the branches, not just the happy path: no answer, busy, voicemail/machine detection,
   guard-blocked, caller hangs up mid-turn, agent-initiated end.
5. Add a "where to look when X" table — the existing `docs/ARCHITECTURE.md` has one and it is the most
   useful thing in that file.
6. Record `post_call_analyzer.py` as dead code in DOC-09.

### Do
- Follow real log lines from TKT-004's validation call. A trace grounded in a real call is worth more
  than one derived from reading code, and you already have the call.
- Name the failure branches. "Call won't connect" is the most common support question there is.

### Don't

## DOC-02 · System architecture overview

**Maps to:** original task 2 · **Est:** 3h · **Status:** ⬜ · **Depends on:** TKT-001, TKT-003

### Goal — done means
`docs/v2/02-architecture.md`: what runs where. Every host, every systemd service, what each does,
what breaks if it stops, how they connect.

### Why this needs TKT-001
The existing `docs/ARCHITECTURE.md` has already drifted — it says Python 3.11 (prod is 3.12) and
describes services binding inside docker-compose (prod is systemd). This doc replaces it, and it can
only be right if it is written from TKT-001's observed process list rather than from the repo's
configuration files. See revisions **R-2**, **R-4**, **R-5**, **R-8**.

### Plan
1. Start from `docs/v2/00-ground-truth.md`. Facts first, then narrative.
2. Host inventory: Hetzner VPS `144.76.17.150` (`Blaze-VoIP-API`) — what runs on it. Vercel — the
   Next.js frontend, auto-deployed from `main`. Managed Postgres and Redis — where they live.
3. Service table — for each: unit name, what it runs, `Type=`, `Restart=`, watchdog, **and what breaks
   if it goes down**. That last column is the one an on-call engineer reads at 3am.

   | Unit | Runs | Type | Restart | Watchdog | If it dies… |
   |---|---|---|---|---|---|
   | `talky-api` | uvicorn, **1 worker** | exec | on-failure | — | everything; note the 1-worker constraint below |
   | `talky-dialer-worker` | dialer | notify | always | 180s | no outbound calls placed |
   | `talky-voice-worker` | voice | notify | always | 180s | |
   | `talky-reminder-worker` | reminders | notify | always | 180s | |
   | `talky-cleanup.timer` | 03:00 nightly | oneshot | — | — | retention stops (currently **dry-run**) |
   | `talky-healthwatch.timer` | every 2 min | oneshot | — | — | worker-death alerting stops |
   | `talky-trunk-status.timer` | every 15s | oneshot | — | — | trunk registration status goes stale |

4. **Document the 1-worker constraint prominently.** `talky-api` runs `--workers 1` because per-call
   ARI state is process-local. Someone will eventually try to scale it by raising that number and
   will break every in-flight call. Say so, in bold, with the reason.
5. Frontend ↔ backend: the Vercel `/api/v1` proxy route, auth via the `talky_at` HttpOnly cookie with
   a flag-gated Bearer fallback, and the WebSocket paths.
6. Data stores: Postgres with RLS, Redis (cache, rate limits, heartbeats, telephony session state).
7. Note the `Admin/` app — a separate Vite/React SPA, not part of the Next.js frontend.
8. Mark `docs/ARCHITECTURE.md` superseded.

### Do
- Write the "if it goes down" column as if for someone with no context. It is the highest-value column.
- State plainly which parts are single points of failure. One VPS, one API process — say it.

### Don't
- ❌ Do not copy the old architecture doc's claims forward. Re-verify each one.
- ❌ Do not describe Kubernetes/Helm (`infra/k8s`, `infra/helm`) as deployed. Mention them as
  unused scaffolding, clearly labelled.
- ❌ Do not describe the monitoring stack as running unless TKT-001 found it running.
