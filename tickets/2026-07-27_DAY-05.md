## TKT-005 · STT connection lifecycle — build (F-01 + F-06)

**Maps to:** listening-path batch 2, increment #9 · **Est:** 6h build (ships Day 06) · **Status:** ⬜

### Goal — done means
A mid-call Deepgram socket drop **reconnects and the call survives**, instead of the call silently
ending. Code complete with unit tests green locally. Deploy is tomorrow (TKT-006) — deliberately a
separate ticket so the change gets a night and a fresh read before it reaches production.

### Why this is here
Today, a dropped STT socket ends the call with no error surfaced to anyone. `deepgram_flux.py` has
reconnect code — cap, backoff, the lot — and **it is unreachable**. This is the single biggest
remaining reliability defect in the voice path, and every "the call just dropped" report is a
candidate symptom.

F-01 and F-06 **must ship in the same commit.** Fixing F-01 makes reconnects actually fire; without
F-06's end-of-turn checkpoint, the replay buffer then re-transmits audio that was already
transcribed, and the caller's last sentence gets spoken into the transcript twice. Shipping F-01
alone converts a call-dropping bug into a transcript-corrupting bug.

**Depends on:** TKT-002 (baseline), TKT-004 (validation call — establishes what "working" sounds
like). **Blocks:** TKT-006, TKT-007, TKT-008.

### Plan
1. **`stt_provider.py`** — add `STTStreamFault(RuntimeError)`. It must compose with, not replace,
   the existing `TerminalSTTError` path. Two distinct meanings: *this stream broke, retry it* vs
   *this provider is unusable, stop*.
2. **`deepgram_flux.py`** — revive the dead reconnect loop, driven by an explicit stop-reason
   vocabulary:
   - unexpected transport close → **reconnect-eligible**
   - auth failure / provider rejection → **fail fast**, never retry (retrying a 401 in a loop is how
     you get rate-limited into a longer outage)
   - clean shutdown → do not reconnect
   - **Reset the stop reason at the start of every attempt.** A sticky reason from attempt 1 that
     survives into attempt 2 is exactly the class of bug that made the original loop unreachable.
3. **`deepgram_nova.py`** — raise on abnormal close, and add a KeepAlive heartbeat.
   ⚠️ **`send_keep_alive()` does not exist in SDK 5.3.0** — this was verified against the installed
   prod version. The correct call is:
   ```python
   await conn.send_control(ListenV1ControlMessage(type="KeepAlive"))
   ```
   Verify against the installed SDK before writing it, not against the SDK's current docs.
4. **`resilient_stt.py`** — re-raise instead of silently returning when there is no secondary
   provider. Silent return is what converts a provider fault into a dead call. Add
   `_ReplayBuffer.checkpoint()` at end-of-turn (this is F-06).
5. Write the tests **before** wiring the reconnect loop live — see the test table.

### Do
- Verify the Deepgram SDK surface against the **installed version** in the prod venv. The design
  note that named `send_keep_alive()` was wrong, caught only by checking. Assume the same about
  anything else the design specifies.
- Keep the two error types semantically distinct and document the distinction in the docstring —
  DOC-08 will reference it.
- Bound the reconnect attempts. An unbounded loop against a provider outage is a self-inflicted
  denial of service on your own API budget.

### Don't
- ❌ **Do not ship F-01 without F-06.** Stated above; it is the main hazard of this ticket.
- ❌ Do not retry on auth or provider-rejection errors.
- ❌ Do not deploy today. Deploy is TKT-006, tomorrow, on purpose.
- ❌ Do not `git stash` in a shared tree — this wiped two engineers' work on 2026-07-21.
- ❌ Do not weaken an existing test to make a new one pass. If a test now fails, either the test
  encoded the old broken behaviour (fix the test, and say so in the commit) or the change is wrong.
