## DOC-06 · Environment variables guide

**Maps to:** original task 6 · **Est:** 4h · **Status:** ⬜

### Goal — done means
`docs/v2/06-environment.md`: every environment variable that matters, explained — what it does, what
breaks if it is wrong or missing, which are critical, where the values come from.

### ⚠️ Re-scoped
The original task says "every `.env` variable". Reality:
- `backend/app/core/config.py` declares ~**37** typed settings on `class Settings`.
- A scan of `backend/app/**/*.py` finds **245 unique env-var names** read ad hoc via `os.getenv()` /
  `os.environ.get()`.

Configuration is decentralised. Explaining 245 variables to the depth the task asks would consume the
week, and most of that effort would land on flags nobody has ever set. So: **three tiers.**

### Plan
1. **Generate the raw inventory first** — a script that walks the AST for `os.getenv` /
   `os.environ.get` / `Settings` fields and emits name, default, and the module it is read in. Commit
   the script to `backend/scripts/`; a generated table that can be regenerated stays true, and one
   that was hand-typed once does not.
2. **Tier 1 — Critical.** Full treatment. If it is wrong or missing, the system does not work or is
   unsafe. At minimum: `DATABASE_URL`, `REDIS_URL`/`REDIS_PASSWORD`, JWT/secret-key material, KMS
   master key, `ASTERISK_ARI_PASSWORD`, the primary provider keys (`DEEPGRAM_API_KEY`, `GROQ_API_KEY`,
   `CARTESIA_API_KEY`), `STRIPE_SECRET_KEY` / `STRIPE_WEBHOOK_SECRET` / `STRIPE_BILLING_DISABLED`,
   `CLEANUP_DRY_RUN`, and every variable `prod_gate.py` checks. For each: purpose, format, what breaks,
   where the value comes from, and whether `prod_gate` enforces it.
3. **Tier 2 — Operational.** One line each, grouped by subsystem (telephony, providers, email, storage,
   security, workers).
4. **Tier 3 — Flags and tuning knobs.** The generated table: name, default, module. Plus one honest
   sentence: *undocumented defaults are the documented behaviour* — these are knobs, and the default is
   what production runs.
5. **Document `prod_gate.py` as the enforcement layer** — what it checks, what it refuses to boot on,
   and why that is a feature (it is what makes a misconfigured production fail loudly instead of
   quietly running in mock mode).
6. Explain the layering: `backend/config/{development,production}.yaml`, `.env`, real environment, and
   which wins.
7. Log the decentralisation itself to DOC-09 as a v2 consolidation candidate.

### Do
- Generate, don't transcribe. 245 hand-copied rows will be wrong within a month.
- Say where each Tier 1 value comes from — which vendor dashboard, which generator script
  (`scripts/generate-secrets.sh`), which teammate. "What breaks if missing" is useless without
  "where do I get one".
- Flag any variable read in more than one place with different defaults. That is a bug, not a doc note.

### Don't
- ❌ **Never print a value.** Names, formats and sources only. Never `cat` a `.env`.
- ❌ Do not commit an example file with real-looking secrets.
- ❌ Do not give Tier 3 the Tier 1 treatment. Rationing depth is the point.

## DOC-07 · Database schema guide

**Maps to:** original task 7 · **Est:** 4h · **Status:** ⬜

### Goal — done means
`docs/v2/07-database.md`: every table explained — what it stores, what writes to it, what reads from
it, which relationships matter, which are high volume.

### ⚠️ Re-scoped
The original task says "all 36 tables". `backend/database/fresh_schema.sql` (5,961 lines) contains
**70** `CREATE TABLE` statements; the older `complete_schema.sql` has 60. Meanwhile
`backend/app/infrastructure/storage/models.py` declares only **4** ORM classes (`Campaign`, `Lead`,
`Call`, `Conversation`) — most of the application talks to Postgres through raw SQL via
`app/core/postgres_adapter.py`.

That gap is itself important: a new engineer who finds the ORM will reasonably assume it is the data
layer, and be wrong about 66 tables.

### Plan
1. Establish which schema file is authoritative, and how it relates to `Alembic/` and the 47 SQL files
   in `backend/database/migrations/`. Two migration mechanisms is a finding — resolve or document it.
2. **Group the tables by domain** and document each group; do not produce a flat alphabetical list.
   Suggested groups: tenancy & identity · campaigns & leads · calls & transcripts · telephony & trunks ·
   security & audit · abuse & call-guard · billing · white-label & partners · webhooks & idempotency ·
   operational.
3. **Full treatment for the tables that matter** — high volume, on the call path, or high consequence:
   `calls`, `call_events`, `stream_events`, `call_legs`, `leads`, `campaigns`, `dialer_jobs`,
   `tenants`/`tenant_users`, `tenant_ai_configs`, `tenant_sip_trunks`, `dnc_entries`, `audit_logs`
   (+ its partitioned variant and `audit_chain_state`). Each: purpose, writers, readers, key
   relationships, volume, retention.
4. One-line coverage for the remainder, with its group.
5. **Document RLS**: which tables carry policies, how `app.current_tenant_id` is set, what
   `app.bypass_rls` is for, and — post-TKT-009 — the `SET LOCAL`-inside-a-transaction rule. Cross-link
   `docs/v2/rls-set-audit.md`. This is the single most important correctness property of the schema.
6. **Document retention**: which tables the cleanup worker touches (`call_events`, `stream_events`,
   `call_legs`), the hardcoded allowlist, and — in bold — that **`calls` is deliberately never in it**,
   and that `CLEANUP_DRY_RUN` is currently `true`.
7. Document the ORM/raw-SQL split explicitly, with a warning.
8. Add a "useful diagnostic queries" section — read-only, the ones an engineer actually needs:
   campaign lead-status counts, `dialer_jobs` by status, recent calls by outcome, trunk registration
   state. VID-18 records against exactly these.

### Do
- Get the diagnostic-query section right. It is the part that will be used weekly.
- Mark high-volume tables clearly. Someone will write a query against `stream_events` without a
  tenant filter, and it should not be their first time hearing that it is large.
- Note foreign-key and cascade behaviour where a delete could cascade somewhere surprising.

### Don't
- ❌ Do not run anything but `SELECT` against production. Use the `dbq` skill.
- ❌ Do not include any real customer data in an example. Redact.
- ❌ Do not present `storage/models.py` as the data layer.
- ❌ Do not document the cleanup worker as active deletion — it is dry-run, and saying otherwise could
  get someone to trust a retention guarantee that is not yet in force.

### Checklist

- [ ] All ~70 tables covered, grouped by domain
- [ ] Full treatment for high-volume / call-path / high-consequence tables
- [ ] RLS documented, cross-linked to the SET-LOCAL audit
- [ ] Retention documented, incl. `calls` never in the allowlist and `CLEANUP_DRY_RUN=true`
- [ ] ORM/raw-SQL split documented with a warning
- [ ] Diagnostic query section (read-only)
- [ ] No customer data in examples
