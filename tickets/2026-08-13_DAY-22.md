## Standing rules for every Sprint 3 recording ticket

Applies to VID-13 … VID-22; not repeated per ticket.

1. **Each session is 30–60 minutes maximum.** If the material does not fit, split it into two clearly
   numbered parts. A 90-minute video does not get watched.
2. **The corresponding Sprint 2 doc is the script.** That is why documentation came first. Before
   recording, read the doc through once and note the section order — the video follows it, so a viewer
   can move between them.
3. **Full narration throughout.** Silent screen-scrolling teaches nothing. Say what you are doing and
   *why* before you do it.
4. **Save to the shared team drive** with the exact title given in the ticket. Consistent titles are
   what make this a library rather than a folder of videos.
5. **Do not skip the complex parts.** The original document is explicit about this, and the complex
   parts are the entire reason the library exists.
6. **Production safety applies while recording.** A live demo on production is still a live action.
   Every "Don't" from Sprint 1 holds: no deletions, no blind restarts, no DB writes, no campaign starts.
7. **Redact before recording, not after.** Close `.env` files, secret managers and vendor dashboards.
   Check the terminal scrollback for keys. A leaked credential in a recording means rotating the
   credential *and* re-recording.
8. **A retake is cheaper than a bad video.** If something goes wrong 10 minutes in, restart.
9. **Each recording is reviewed by one other person before 🟢** — ground rule 5 applies to videos too.

## VID-13 · Codebase walkthrough

**Maps to:** original task 13 · **Est:** 5h (prep + record + review) · **Status:** ⬜
**Script:** DOC-02 (architecture) + DOC-01 (call flow)

### Goal — done means
Two recordings — **Part 1: Backend** and **Part 2: Frontend** — screen-sharing the full folder
structure, explaining what every major folder, file and module does.

### Why two parts
The backend alone is `api/` (~55 routers), `core/`, `domain/` (interfaces, models, repositories,
services), `infrastructure/` (10 adapter families), `services/`, `utils/`, `workers/`, plus a
27-file voice pipeline package. The frontend is a separate Next.js app with ~28 route groups, a
proxy layer and its own auth. One hour cannot cover both without skimming, and skimming the complex
parts is exactly what rule 5 forbids.

### Part 1 — Backend (~50 min)
1. `backend/app` top level, and **why the layering exists**: `domain/` has no I/O, `infrastructure/`
   talks to the outside world, `services/` glues, `api/` is the HTTP edge. Mention that
   `tests/unit/test_no_domain_api_imports.py` enforces the domain→api direction with an AST test and
   two allowlisted residuals.
2. `api/v1/endpoints/` — the ~55 routers, the ones that matter (`campaigns`, `calls`,
   `telephony_bridge`, `telephony_sip/`, `auth`, `admin`), and how routing is assembled.
3. `domain/services/` — the call path: `telephony/` (post-split modules), `voice_orchestrator/`
   (post-split), `voice_pipeline/` (walk the 27 files at least by group), `call_guard/` (post-split),
   `dialer/`, `call_summary/`.
4. `infrastructure/` — the adapter families: `stt/`, `llm/`, `tts/`, `realtime/`, `telephony/`,
   `connectors/`, `providers/`, `storage/`, `assistant/`, `metrics/`.
5. `workers/` — the four workers and their systemd units.
6. `core/` — config, DB pool, security, middleware stack, `prod_gate`.
7. `tests/` — 292 test files across unit / security / integration / chaos / load, and how to run the gate.

### Part 2 — Frontend (~40 min)
1. `Talk-Leee/` — Next.js 15 / React 19, the App Router layout, the ~28 route groups.
2. `src/lib/` — the API client family (`backend-api.ts` and the split-purpose clients) and the
   `src/app/api/v1/[...path]/route.ts` proxy.
3. Auth — `middleware.ts`, `auth-token.ts`, the `talky_at` HttpOnly cookie, the flag-gated Bearer
   fallback and where it is legitimately used.
4. The key screens: campaigns, contacts, AI Options, calls, recordings, billing.
5. `Admin/` — call out that it is a **separate Vite/React SPA**, not part of this app.

### Do
- Explain *why* a folder exists before listing what is in it.
- Open a real file in each layer and read a few lines aloud. Structure without a sample is abstract.
- Say plainly where the code is messy. A walkthrough that presents everything as clean is not trusted.

### Don't
- ❌ Do not read directory listings aloud without explanation.
- ❌ Do not skip `voice_pipeline/` because it is large — it is the product.
- ❌ Do not record with a `.env` or vendor dashboard open in another tab.
- ❌ Do not describe the pre-split module layout. Record against the frozen post-split code.
