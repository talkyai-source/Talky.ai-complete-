## TKT-001 · Production ground-truth audit (read-only)

**Maps to:** prerequisite for original tasks 2, 3, 4, 10, 21 · **Est:** 3h · **Status:** ⬜

### Goal — done means
A written, evidence-backed answer to six questions, committed as `docs/v2/00-ground-truth.md`,
each answer accompanied by the command that produced it and its raw output:

1. Which systemd units exist on the box and which are `active`? (Compare against the seven units in
   the repo — is `talky-voice-gateway` real?)
2. Is the C++ voice gateway process running? If yes, from which unit file, and is that unit file in
   version control?
3. Which telephony processes are actually running — Asterisk only, or also
   OpenSIPS / Kamailio / RTPengine / FreeSWITCH?
4. Is anything Docker running on the box at all?
5. Is there a Prometheus, Grafana, Alertmanager or exporter process running?
6. Does the server working tree match `origin/main`, with only the expected drift?

### Why this is here
The V2 document assumes a five-component telephony stack, a C++ gateway, and a Grafana/Prometheus
monitoring stack. The repo contains configuration for all of those, but repo configuration is not
evidence that something runs — see revisions **R-4**, **R-5**, **R-8**. Documenting a standby
component as active is precisely the failure mode this whole project exists to eliminate.

**Depends on:** nothing. **Blocks:** DOC-02, DOC-03, DOC-04, DOC-10, VID-17, VID-21, TKT-003.

### Plan
1. SSH in read-only. Sudo password comes from the user — it is deliberately in no repo file.
2. Run the command set below, capturing raw output verbatim into a scratch file as you go.
3. Write `docs/v2/00-ground-truth.md`: one section per question, each with command + output + a
   one-sentence conclusion.
4. For every mismatch found between repo and reality, add a line to the DOC-09 running findings file.
5. If `talky-voice-gateway` is running from an unversioned unit file, **copy the unit file content
   into the ticket and raise it immediately** — that is a live production risk, not a documentation
   detail.

```bash
# inventory — all read-only
systemctl list-units 'talky-*' --all --no-pager
systemctl list-timers --all --no-pager
systemctl is-active talky-api talky-dialer-worker talky-voice-worker talky-reminder-worker
systemctl cat talky-voice-gateway 2>&1 | head -40      # exists? or "No files found"?
ps -eo pid,comm,args --sort=-%mem | head -40
pgrep -a -f 'asterisk|opensips|kamailio|rtpengine|freeswitch' || echo "NONE RUNNING"
pgrep -a -f 'voice_gateway' || echo "GATEWAY NOT RUNNING"
pgrep -a -f 'prometheus|grafana|alertmanager|node_exporter|redis_exporter' || echo "NO MONITORING STACK"
docker ps -a 2>&1 | head -20 || echo "docker not installed"
cd /opt/talky && git log -1 --format='%h %s' && git status --porcelain
curl -s -o /dev/null -w '%{http_code}\n' http://localhost:8000/api/v1/healthz/deep
curl -s http://localhost:8000/api/v1/healthz/workers
```

### Do
- Capture raw output. Paste it into the doc. An assertion without its command is not a finding.
- Poll `/healthz/deep` a few times if the first call returns `000` — that is a known pre-bind race,
  not a failure.
- Record what is **absent** as explicitly as what is present. "No monitoring stack is running" is
  one of the most valuable sentences this ticket can produce.

### Don't
- ❌ Do not restart, reload, enable, disable or `daemon-reload` anything. This is read-only.
- ❌ Do not `cat` `/opt/talky/backend/.env`, or any file under `secrets/`.
- ❌ Do not run `git pull`, `git checkout`, `git stash`, `git clean` or `git reset` on the server.
- ❌ Do not write to the database. Read-only `SELECT` via the `dbq` skill only.
- ❌ Do not "fix" anything you find today. Today is observation. Findings become tickets.

## TKT-002 · Baseline freeze — record the starting state

**Maps to:** prerequisite for original task 36 · **Est:** 1.5h · **Status:** ⬜

### Goal — done means
A recorded, reproducible baseline that every later claim of "no regression" can be measured against:
the exact test-gate numbers, the prod commit, and an annotated tag.

### Why this is here
Original task 36 asks for "pytest tests/ shows 0 failures after all fixes". That is unverifiable
without knowing the number before. Sprint 1 is about to refactor 5,230 lines of live call-path code
across three files — a baseline is the only thing that distinguishes "my split broke a test" from
"that test was already failing".

**Depends on:** nothing. **Blocks:** TKT-017 (code freeze), REV-39.

### Plan
1. Run the full gate locally and record the exact line:
   ```bash
   cd backend && python -m pytest tests/unit tests/security -q
   ```
   Last known good: **3,547 passed, 6 skipped, 0 collection errors**.
2. If the count differs, find out why before proceeding. A changed baseline is a finding.
3. Re-run `tests/unit/test_resilient_providers.py::test_stt_ask_ai_singleton_respects_recovery_window`
   five times in isolation. It is a known flake (50 ms timing window, fails only under full-suite CPU
   load). Record the isolated result — it is fixed for real in TKT-013.
4. Record prod HEAD and tag it:
   ```bash
   git rev-parse --short origin/main
   git tag -a v1-baseline-2026-07-23 -m "Pre-V2-readiness baseline: <N> passed, prod <hash>"
   ```
   Do **not** push the tag until the numbers are in it.
5. Write `tickets/BASELINE.md` with: gate numbers, prod commit, rollback commit (`4693bf13`),
   file line-counts for the three split targets, and the current `/healthz/workers` output.

### Do
- Record numbers, not impressions. "Tests pass" is not a baseline; "3,547 passed, 6 skipped, 0 errors
  in 152s" is.
- Capture the three god-file line counts now — they are the acceptance criteria for TKT-014/015/016.

### Don't
- ❌ Do not fix failing tests today. Record them. Fixing changes the baseline you are trying to take.
- ❌ Do not use `--continue-on-collection-errors`. If you need it, something regressed and that is
  the finding.
- ❌ Do not tag until the numbers are known and written into the tag message.
