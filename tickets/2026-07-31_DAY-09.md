## TKT-012 · Stripe live mode — readiness (fix 9)

**Maps to:** original task 27 (fix 9; fix 10 is already done) · **Est:** 4h · **Status:** ⬜

### Goal — done means
Everything that can be prepared and rehearsed for Stripe go-live is done and proven in **test mode**,
so that the actual go-live is a key swap plus a webhook registration — not a discovery exercise.

### Why this is here
Billing is **structurally mock** today. `BillingService._should_use_mock_mode()` returns true when no
key is present, and no `STRIPE_SECRET_KEY` is set. The SDK was pinned in `requirements.txt` on
2026-07-22 (commit `80f6cdab`, `stripe>=12.0.0,<15.0.0` — capped below v15 because of a
`StripeObject` dict-inheritance break; there is a flagged `.get()` call in `billing_service` to audit).
`prod_gate.py` already refuses to boot on a non-live key or a missing SDK — that is fix 10, done.

**The live key is the user's to provide.** This ticket does everything that does not require it.

**Depends on:** user provides `sk_live_` (for the final step only). **Blocks:** nothing.

### Plan
1. Confirm the SDK installs cleanly in the prod venv version (Python 3.12, not local 3.11).
2. Audit the flagged `billing_service` `.get()` call against the pinned SDK version's `StripeObject`
   behaviour. This is exactly the kind of thing that works in test mode and breaks on a live payload.
3. **Rehearse the whole flow locally in test mode** (`sk_test_`): checkout → webhook → subscription
   status written to the DB. This must be local — `prod_gate` now refuses a test key on production.
4. **Verify the webhook path — this is the classic go-live breaker.** Stripe requires HTTPS, and
   signature verification needs the **raw request body byte-for-byte**. The frontend proxies
   `/api/v1` through Vercel (`Talk-Leee/src/app/api/v1/[...path]/route.ts`). Prove the raw body
   survives that proxy. If it does not, expose a direct TLS endpoint for webhooks only.
5. Map plans to live Price IDs in `plans.py`. **Test-mode price IDs do not work in live mode** —
   live Products/Prices must be created via the API.
6. Write `docs/v2/stripe-golive-runbook.md` — the exact ordered steps for the day the key arrives.
7. **Stop there.** Do not set the live key today.

### Do
- Prove the raw body survives the Vercel proxy before anything else. If it does not, every other step
  is wasted and you want to know on day 9, not on go-live day.
- Confirm the `STRIPE_BILLING_DISABLED` kill-switch works before you need it.
- Write the runbook as if someone else will execute it. They might.

### Don't
- ❌ Do not set a live key in production today. `prod_gate` refuses a test key on prod, so a
  half-configured attempt takes the API down on a Friday.
- ❌ Do not take a real payment as a test. That is a go-live-day step (cheapest plan, then refund).
- ❌ Do not commit any key, test or live, in any form.
- ❌ Do not disable `prod_gate`'s Stripe checks to make boot succeed. They are the fix.

### Checklist
- [ ] SDK installs in the prod venv Python version
- [ ] `billing_service` `.get()` call audited against the pinned SDK
- [ ] Full test-mode flow rehearsed locally: checkout → webhook → DB write
- [ ] **Raw-body integrity through the Vercel proxy proven** (or a direct endpoint designed)
- [ ] Live Product/Price creation scripted; `plans.py` mapping reviewed
- [ ] `STRIPE_BILLING_DISABLED` kill-switch verified
- [ ] `docs/v2/stripe-golive-runbook.md` written
- [ ] Live key **not** set — go-live is a separate, scheduled, non-Friday event
- [ ] Peer-reviewed

## TKT-013 · Debt sweep

**Maps to:** supports original tasks 33 and 36 · **Est:** 3h · **Status:** ⬜

### Goal — done means
Six known, small, individually-annoying debt items are closed, so that none of them shows up as
noise during Sprint 2's documentation work or Sprint 4's verification.

### Why this is here
Each of these is already diagnosed. Left alone, each will be rediscovered — and worse, each will look
like a *new* problem to whoever writes the docs.

| Item | What | Risk if left |
|---|---|---|
| 1 | Apply migration `20260717_calls_lead_id_nullable.sql` | Schema drift between code and DB |
| 2 | `app/infrastructure/storage/models.py` still says `nullable=False` for the same column | Model contradicts the DB |
| 3 | Delete the stale root `ci.yml` | It already misled one external audit (R-13) |
| 4 | Fix the flaky `test_stt_ask_ai_singleton_respects_recovery_window` | Undermines every "gate is green" claim |
| 5 | Vectorize `audio_utils.pcm_to_alaw` / `alaw_to_pcm` | Per-sample Python loops on the audio path |
| 6 | External uptime monitor on `/healthz/workers` | Nothing currently catches "the whole box died" |

**Depends on:** item 1 needs the **user** to run the migration — the safety classifier blocks remote
DB writes, and that is correct behaviour, not an obstacle to route around.

### Plan
1. **Migration:** prepare the exact `psql` command, hand it to the user, confirm applied, then fix
   `models.py` to match. Do not attempt the write yourself.
2. **Stale `ci.yml`:** delete the root file. Confirm first that nothing references it —
   `.github/workflows/ci.yml` is the live one and its `secret-guard` job explicitly excludes both
   paths from its own scan, so read that job before deleting.
3. **Flaky test:** the 50 ms recovery window is too tight under full-suite CPU load. Widen it, or
   inject a clock. Do **not** mark it skip — that hides a real timing assumption.
4. **A-law:** apply the same numpy-LUT treatment µ-law already received, with the same **exhaustive
   bit-exactness test** across all 256 values against the scalar reference. Bit-identical or it does
   not ship.
5. **Uptime monitor:** point an external check at `/healthz/workers`. Journald alerting is covered by
   `talky-healthwatch.timer`, but that timer dies with the box it runs on.

### Do
- Prove A-law bit-exactness exhaustively. µ-law set the precedent; audio correctness is not a place
  for spot checks.
- Fix the flake properly. A test suite you have to caveat is a test suite nobody trusts.

### Don't
- ❌ Do not run the migration remotely yourself. Hand the command to the user.
- ❌ Do not `skip` the flaky test.
- ❌ Do not delete root `ci.yml` before confirming nothing reads it.
- ❌ Do not batch all six into one commit. Six small commits; each is independently revertible.
