## DOC-05 · Audio pipeline guide

**Maps to:** original task 5 · **Est:** 4h · **Status:** ⬜ · **Depends on:** DOC-01

### Goal — done means
`docs/v2/05-audio-pipeline.md`: the complete conversion chain from TTS output to the caller's ear and
back, every format conversion, every sample-rate change, every codec step, with code references.

### Why this matters more than it looks
Audio format bugs are the hardest class of bug on this project to diagnose, because they present as
"it sounds bad" rather than as an exception. Two have already shipped and been fixed: a near-zero
µ-law buzz on egress, and an underrun buzz caused by event-loop blocking in DB teardown. A third —
the sample-rate "fix" proposed in the V2 document — would have **caused** distortion rather than
fixed it (revision **R-10**).

### ⚠️ Established facts to build on, not re-litigate
- Production runs one uniform TTS format: **`s16le@16000`**. 655 of 655 format probes across 14 days
  agree. Zero mismatch warnings.
- The gateway self-reports **`internal=16000Hz, wire=8000Hz PCMU`**.
- **Do not "align" the sample rates to 8000.** The V2 document's fixes 1–3 are refuted with production
  evidence and are dropped from this plan permanently.

### Plan
1. Map the **egress** chain: LLM text → sentence segmentation → TTS provider → PCM → egress hygiene →
   resample 16k→8k → µ-law encode → RTP → carrier → caller.
2. Map the **ingress** chain: caller → carrier → RTP → µ-law decode → resample 8k→16k → STT provider →
   transcript chunks → turn logic.
3. For every conversion point, name the function. Central module is `app/utils/audio_utils.py`:
   `resample_audio()` (soxr), `pcm_to_ulaw()` / `ulaw_to_pcm()` (vectorized numpy LUTs),
   `pcm_to_alaw()` / `alaw_to_pcm()` (vectorized in TKT-013), and the combined format→resample→encode
   path. Also `infrastructure/telephony/telephony_media_gateway.py`,
   `twilio_media_gateway.py`, `egress_audio_hygiene.py`, and the realtime adapters.
4. Document the **realtime path separately** — OpenAI Realtime uses g711 µ-law at 8k end to end, so it
   bypasses resampling entirely. Two different audio topologies exist; conflating them is a trap.
5. Document `egress_audio_hygiene.clean_egress_pcm16` and the near-zero-sample buzz it prevents.
6. Document soxr: which version, why it rather than a naive resampler, and what breaks without it.
7. **Add a diagnostic section**: how to tell, from symptoms, which stage is wrong. Buzz vs. chipmunk
   vs. slow-motion vs. silence vs. one-way audio each point at a different, identifiable stage. This
   is the section that will actually get used.

### Do
- State the canonical rates once, prominently, and reference that statement everywhere else.
- Write the symptom→stage table. It is the highest-value part of this doc.
- Note that `audioop` was removed in Python 3.13 — prod is on 3.12, so this is a live migration
  constraint, not trivia.

### Don't
- ❌ Do not repeat the "sample rates are misaligned" claim. It is false and it has already cost time.
- ❌ Do not conflate the cascaded and realtime paths.
- ❌ Do not use line numbers.

## DOC-08 · AI provider configuration

**Maps to:** original task 8 · **Est:** 3h · **Status:** ⬜

### Goal — done means
`docs/v2/08-ai-providers.md`: how to switch providers, what the fallback chain is, what each API key
controls, how to add a new provider.

### Plan
1. **Inventory by role.**
   - STT: `infrastructure/stt/` — `deepgram_flux.py` (primary; turn-detection model),
     `deepgram_nova.py`, `deepgram.py`, `factory.py`
   - LLM: `infrastructure/llm/` — `groq.py` (primary), `gemini.py`, `factory.py`
   - TTS: `infrastructure/tts/` — `cartesia.py`, `elevenlabs_tts.py`, `deepgram_tts.py`,
     `google_tts.py` / `google_tts_streaming.py`, plus `elevenlabs_clone.py` / `elevenlabs_catalog.py`,
     `factory.py`
   - Realtime (combined): `infrastructure/realtime/` — `openai_realtime.py`, `xai_realtime.py`
2. **Fallback chain.** `resilient_stt.py`, `resilient_llm.py`, `resilient_tts.py` — what triggers a
   failover, what happens with no secondary configured (changed in TKT-005: it now re-raises rather
   than silently returning), and the realtime→cascaded mid-call fallback added by fix 14.
3. **Key management.** `infrastructure/providers/key_pool.py` (rotation across multiple keys — note
   the plural env vars, e.g. `DEEPGRAM_API_KEYS`) and `provider_concurrency.py` (per-provider caps —
   now honoured by Flux as well as Nova after TKT-007).
4. **Switching providers** — per tenant via `tenant_ai_configs`, per campaign via AI Options. Document
   both, and which wins.
5. **Adding a new provider** — the interface in `domain/interfaces/` to implement, the factory to
   register with, the config to add, the tests to write. A worked checklist.
6. **Model selection and known quirks.** Curated menu; ~0.5 temperature; compliance-floor plus
   per-model addendum prompt architecture. Record explicitly that **Qwen 3.6 27B was withdrawn once
   (2026-06-27) for evading the AI-disclosure question and hallucinating** — a model that fails the
   disclosure question is a compliance problem, not a quality preference.
7. Note that `backend/config/providers.yaml` exists and how it relates to env-var configuration.

### Do
- Document per-model quirks. They are real operational knowledge and they live nowhere else.
- Make the "add a provider" section a checklist someone can follow, not prose.
- State which config layer wins when tenant and campaign disagree. Someone will hit it.

### Don't
- ❌ Do not print any API key, or any fragment of one.
- ❌ Do not document a provider as available without checking its adapter is wired into its factory.
- ❌ Do not recommend a model without noting its known failure modes.
