## VID-17 · Telephony configuration walkthrough

**Maps to:** original task 17 · **Est:** 4h · **Status:** ⬜ · **Script:** DOC-04

### Goal — done means
A recording explaining the live telephony configuration on screen: what each setting does, what breaks
if it is changed, and how to add a SIP trunk.

### ⚠️ Scope corrected from the original task
Task 17 says *"explain the Asterisk and FreeSWITCH config files live"*. Per revision **R-5** and
TKT-001's process list, **FreeSWITCH is not running in production** and its config directories are
almost entirely `.gitkeep` placeholders. Recording a walkthrough of empty scaffold config as if it
were production would teach something false.

**What to record instead:** the **live** Asterisk configuration in depth, then a short, clearly
labelled section explaining what the FreeSWITCH / OpenSIPS / Kamailio / RTPengine scaffold is, why it
exists, and that it is not deployed.

### Plan
1. Open with the process list — *this is what is actually running.* Ninety seconds, and it prevents
   every later misunderstanding.
2. **Asterisk configuration**, file by file, from the live box (read-only):
   - `pjsip.conf` — the trunk to the carrier, registration, auth, endpoint/AOR structure
   - `extensions.conf` — the dialplan and how a call reaches the Stasis app `talky_ai`
   - `ari.conf` / `http.conf` — the ARI interface the backend drives, bound to `127.0.0.1:8088`
   - `rtp.conf` — the port range and why it must match the firewall rules
3. Note the **two-config situation**: `setup-asterisk.sh` writes `/etc/asterisk/*` directly, while
   `telephony/asterisk/conf/*` also exists in the repo. Say clearly which one is authoritative.
4. **Reading SIP state live**: `pjsip show registrations`, `pjsip show endpoints`, and turning the SIP
   logger on — **and off again**, on camera, so the habit is transmitted.
5. **How to add a SIP trunk** — end to end: carrier details → PJSIP config → reload → verify
   registration → the application-side trunk record → caller-ID verification. Emphasise the
   caller-ID trap: the real caller-ID for `blaze-pool-150004` is `+442046132301`, and a one-digit-short
   value has already caused a live failure.
6. **What breaks if changed** — the highest-value section. Port ranges versus firewall rules, codec
   settings, the ARI password versus the backend `.env`, registration intervals.
7. Short labelled section: the `telephony/` scaffold — what it is, the three-phase design work behind
   it, and that it is not deployed.

### Do
- Show a real registration in both states — registered and, if safely reproducible, failed. Reading
  logs is a skill best taught with a failure on screen.
- Turn the SIP logger off on camera. Leaving it on floods the journal.

### Don't
- ❌ **Do not modify any Asterisk config on production.** This is read-only. Demonstrate "how to add a
  trunk" as a walkthrough of the steps and files, not by adding one.
- ❌ Do not show carrier credentials. `setup-asterisk.sh` contains hardcoded credentials — reference
  the file, never display it.
- ❌ Do not reload or restart Asterisk during the recording.
- ❌ Do not present the scaffold as production.

## VID-18 · Database walkthrough

**Maps to:** original task 18 · **Est:** 3h · **Status:** ⬜ · **Script:** DOC-07

### Goal — done means
A recording opening the production database and walking through the key tables live: what the data
looks like, how to run diagnostic queries, what to look for.

### ⚠️ Safety framing — read before starting
The original task says "open production DB in pgAdmin". This is a **live multi-tenant database with
real customer data**. This recording is **read-only, `SELECT` only**, and every query is stated aloud
before it is run.

Two additional constraints:
- **Customer data will be on screen.** Names, phone numbers, transcripts. Either use a tenant the user
  designates as safe for recording, or blur/limit the result sets. Decide this **before** recording,
  and state the choice on camera so viewers understand why the data looks the way it does.
- The safest approach is `LIMIT` plus deliberate column selection — never `SELECT *` on `leads` or
  `calls`.

### Plan
1. Connect read-only. Show the connection method and state the read-only constraint on camera.
2. Schema tour by domain group, following DOC-07: tenancy · campaigns & leads · calls & transcripts ·
   telephony & trunks · security & audit · billing. Roughly 70 tables — cover the groups, detail the
   important ones.
3. **RLS demonstrated live** — this is the most important part. Show `app.current_tenant_id`, show
   that a query without it returns nothing, and explain `app.bypass_rls` and the `SET LOCAL`-in-a-
   transaction rule from TKT-009.
4. **Diagnostic queries** from DOC-07, run live, each explained before execution:
   - campaign lead-status counts — "why is this campaign not dialling?"
   - `dialer_jobs` by status
   - recent calls by outcome
   - trunk registration state
5. **Volume awareness** — point out the high-volume tables (`call_events`, `stream_events`) and say
   plainly what an unfiltered query against them would do.
6. **Retention** — which three tables cleanup touches, that `calls` is deliberately never in the
   allowlist, and that `CLEANUP_DRY_RUN` is currently `true`, so nothing is being deleted yet.

### Do
- State every query aloud before running it. That habit is what the video teaches, beyond the queries
  themselves.
- Show the RLS demonstration properly. It is the single most important property of this schema and it
  is invisible in a schema diagram.

### Don't
- ❌ **No writes. No `INSERT`, `UPDATE`, `DELETE`, `DROP`, `ALTER`, `TRUNCATE` — not even in a
  transaction you intend to roll back.** Not on camera, not off it.
- ❌ No `SELECT *` on tables holding personal data.
- ❌ Do not show the connection string or any credential.
- ❌ Do not run an unfiltered query against a high-volume table to "show how big it is". Use
  `pg_class` estimates.
